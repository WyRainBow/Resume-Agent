# Resume Diagnosis

一次诊断同时产出诊断结论和只读修改建议。本轮只分析，不修改简历，不调用编辑工具，不启动自动优化。

## 固定检查顺序

1. 结构完整度：基本求职方向、教育、经历、项目、技能、时间线。
2. 成果证据：动作、技术手段、可核验结果、业务影响。
3. 面试风险：技能与经历能否互相印证、时间是否一致、表述是否可能引发但无法回答的追问。
4. 岗位匹配：目标方向、经历和技能之间是否形成对应关系。

## 六秒门

- 只看简历顶部（基本信息 + 排在最前的内容模块）六秒：能否看出目标角色、最强技术栈/领域、至少一个量化成果。
- 看不出 → 计入 should_fix（连目标角色/任何成果都看不出时升 must_fix），并在结论里点明应把最强证据前置。

## 技术岗从严标准（条件启用）

从简历内容判断求职方向为技术岗（研发/算法/数据/测试/运维/客户端等）时，按大厂技术简历筛选口径从严：

1. 技能栏必须标注掌握程度分级（熟练掌握/熟悉/了解），并落到框架与深度；只写「熟悉 Java」这类笼统表述 → 计入 should_fix。
2. 项目经历要有能支撑面试追问链（动机→方案→难点→数据效果→收获）的技术亮点；只有职责罗列、无亮点 → 计入 must_fix。
3. 成果量化必须落到具体业务/性能指标（QPS、耗时、DAU、留存、转化率、研发周期等）；只写「参与/负责」而无量化 → 视为未量化。
4. 表述不清（读不懂做了什么）是硬淘汰项 → 一律 must_fix。
5. 缺知名公司实习时，竞赛/奖学金/学生干部经历可作补位证据，应在优势里点出。

非技术岗简历不套用本节标准。

## 证据纪律

- 每个结论引用简历中的具体事实或明确缺失，禁止通用套话。
- 不复述姓名、电话、邮箱、头像地址和精确住址。
- 不把启发式分数说成真实初筛概率或面试通过率。
- 公开 thought 只说明当前判断、下一动作和验证目标；详细证据进入诊断卡，不输出 raw reasoning。
- 工具调用完成后不再写一份重复的 Markdown 诊断报告。

## 输出纪律

- 返回结构化评估、诊断报告和修改建议。
- 报告与建议必须共享同一个 assessment_id 和 resume revision。
- 诊断完成后停止，明确告知本轮没有修改简历。
