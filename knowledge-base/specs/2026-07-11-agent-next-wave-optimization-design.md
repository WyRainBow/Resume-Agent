## 一、执行摘要(先看这个)

**结论一句话**:Wave 2a 的减法(LLM-first 了断)执行得比初步判断以为的还要彻底——审计报告"优先级 3 件事"**全部落地,包括原本被标注为"暂不派"的 Task C**。当前代码已经非常接近调研报告描述的"理想垂直 Agent"(几个清晰工具 + 一份好 prompt + LLM 自主编排)。接下来**不存在需要动架构手术的任务**,真正值得排的只有两件小事,其余两个方向应明确暂缓、不硬造任务。

**核实后对四方向的定性**:

| 初步方向 | 初步判断 | 核实后结论 | 变化 |
|---|---|---|---|
| 一、文件级决策(IntentRouter/ToolInvocationBuilder) | 悬着没定,需评估 | **成立且可立即做**。给出明确方案:`ToolInvocationBuilder` 内联回收,`IntentRouter` 保留但收尾冻结的注入脚手架 | 从"悬着"→"给出确定方案" |

## 二、对四个初步方向的逐条核实与修正

### 方向一:文件级决策 —— 成立,且现在能给出确定方案

**核实到的现状**(读代码而非采信):

- `intent_router.py` 119 行、`tool_invocation_builder.py` 54 行,均确认。
- `ToolInvocationBuilder` 确实**只剩 `build_apply_optimization` 一个方法**,且**只有一个调用点**:`manus.py:714` 的 `_handle_optimize_confirm`。为它保留一个独立文件 + `ToolInvocation` dataclass,属于 CLAUDE.md §2.2「不为单次使用场景引入抽象」的反面。
- `IntentRouter` **不薄**——它承载了 LLM-first 的唯一契约:每轮恰好调用一次 `process_input`、统一让权判定、`enhanced_query` 清洗(规则改写一律丢弃,防止"名义让权、实际遥控")。这是个真接缝,值得独立文件。
- **但发现一处冻结的中途状态**:让权守卫函数 `_has_send_email_intent` / `_looks_like_compound_request` / `_rule_intent_yield_reason` 仍留在 `manus.py` 模块级(51/67/76 行),通过 `yield_reason_fn=` / `compound_request_fn=` **注入**给 `IntentRouter`。`intent_router.py:56-57` 注释和审计报告 §3.D 都写明:注入是"S4c 收口后可平移进本模块"的临时脚手架,为保重构期行为等价而搭。**LLM-first 已了断、行为等价约束已消失,这个"以后再搬"至今没搬**——是重构中途状态的凝固。

**结论**:方向成立。文件级决策不该继续"悬着",具体方案见 **§四**。

### 方向二:灵魂/人设 —— 需修正,"一项都没落地"过于严厉

逐字读完 `prompt/manus.py` 的 `SYSTEM_PROMPT`(19-219 行)+ `greeting.py`,对照调研报告 §4 的标准核实,得到比初步判断更细的结论:

**已经隐含做到的(初步判断漏算了)**:

1. **人设身份存在**:开篇 "你是 coco，一个专业的 AI 简历助手"(19 行)——身份有,但只有一行,是"名字 + 职能",不是"性格 + 信念"。
2. **交互层语气已分层且放对了位置**:`GREETING_FAST_PATH_PROMPT`(greeting.py:20)"像一个真的会聊天的朋友那样回应,不要像客服"——这正是调研报告 §4.2 说的"人设放交互层"的正确实践,且它只影响问候通道的表达,不碰决策内核。
3. **反编造价值观已隐含存在,只是以"禁令"形式散落**:"严禁凭空编造占位简历"(76)、"不能凭空创造"(93/117)、`OPTIMIZE_SECTION_LLM_ADDENDUM` 的"缺数字时不编造""禁止空话与重复"(274-275)。今天实测"LLM 主动拒绝编造数据、反问用户"这个好行为,**根源就是这批散落的禁令**——所以"价值观"其实部分生效了,只是没被当成价值观来组织。
4. **人设不污染安全内核的纪律已守住**:只读拦截在 `execute_tool`(manus.py:242-257,确定性 Python)、admin 才注册发邮件工具(202-203)、发送语义让权守卫(47-83)——这些决策/安全内核都在代码里,与 prompt 里的人设文本物理分离。符合调研报告 §4.2「个性不碰安全规则/权限/核心决策」。

**真正还没做的**:

1. **"规则味重于价值观味"**:整份 prompt ~200 行里,压倒性多数是操作规则清单(场景→工具表、黄金规则、只读规则、path 对照表、富文本约束、诊断输出模板)。审美判断是以"禁止 X"的否定式散落各处,而非调研报告 §4.1 说的正向价值观("我坚信简历要有可量化结果、动词开头…")。
2. **无稳定人设核心块 / 无 SOUL.md 式渐进披露**:人设散在开篇一行 + 各 fast-path fallback 文案里,没有一个"coco 的身份 + 简历价值观 + 边界"的稳定核心块;`PromptBuilder` + `_skills_cache` 的渐进披露基建部分存在,但没用来承载人设。

**重要的判断分寸**(防止过度改造):prompt 里**大部分规则不该被"价值观化"**——HTML 富文本格式、path 叶子字段对照、`%%SUGGESTIONS%%` 协议,这些是 ACI(Agent-Computer Interface)确定性契约,是 LLM 必须精确遵守的格式知识,把它们改写成"价值观"只会让输出格式失控。**值得价值观化的只有"审美判断"那一小块**(什么是好简历、什么时候该主动提意见)。这是一次**精准收敛**,不是"重写 system prompt"。

改进方向的对比示例见 **§五**。

### 方向三:验收契约 / evaluator —— 成立,维持暂缓

初步判断"先不预判性加,等观察到质量问题再引入独立 evaluator"完全正确,维持不变。补充两点核实:

- 删完规则引擎后,`think()` 里对"什么算优化完成/编辑成功"确实没有语义质量校验层(只有 `execute_tool` 的只读拦截、`experience_entry.py` 的字段/HTML 规范化这类确定性校验)。这是事实。
- **但今天实测的"LLM 主动拒绝编造数据"恰恰是"当前不需要 evaluator"的证据**——模型基线已经能守住质量底线。调研报告 §四.3(2026 补充)明确:"evaluator 角色按需引入,只在任务超出模型基线能力时才值得加,模型本身能做好就别硬加"。现在硬加一个 evaluator,既是投机性设计(CLAUDE.md §2.2),又会踩调研报告 §2.3 Reflexion 的"自评重犯自己的错误"坑(若不做 generator/evaluator 分离)。
- **触发条件**(满足任一再考虑立项):① 真实使用中反复观测到 LLM 瞎改 / 幻觉数字 / 优化后质量下降;② 攒到一个 ~100 条 golden 的小 eval 集能量化出质量缺口。届时必须遵守 generator/evaluator 分离。

### 方向四:垂直合规(一致性校验 + 反 AI 味)—— 成立,优先级下调

- 这是 2026 补充调研引入的**纯新功能**(招聘方在查 AI 简历的日期重叠/不可能晋升),不是架构清理。
- 诚实评估:**这是四个方向里最不该现在做的**。理由:① 零用户/产品信号,纯外部趋势驱动;② 是一个独立检测子系统(一致性规则 + 反 AI 味模型/prompt),体量远大于其余任务;③ 与"先把窄工作流做深"的垂直 Agent 纪律(调研 §4.5)一致的做法是——等核心优化体验打磨稳了再谈扩边界。
- **触发条件**:产品侧明确决定把"简历合规/防一眼假"作为卖点或需求,且走完 `/brainstorming`。在此之前不排任何实现任务。

## 四、方向一:文件级决策的具体建议

不含糊,给结论。三个候选方案对比:

| 方案 | 做法 | 优点 | 缺点 | 采纳 |
|---|---|---|---|---|
| A 保持现状(两文件都独立 + 注入) | 什么都不动 | 零改动 | 冻结的注入脚手架永久留存;54 行文件为单次使用的构造器续命,误导后来者以为还会长新构造器 | ✗ |
| B 内联 ToolInvocationBuilder + 平移守卫进 IntentRouter | ToolInvocationBuilder 删除内联回 manus;守卫函数搬进 IntentRouter,去掉注入;IntentRouter 保留独立 | 消灭单次使用抽象;收尾"以后再搬"的中途状态;IntentRouter 作为 LLM-first 唯一契约的真接缝独立留存,职责清晰 | 需同步改两个测试文件 | **✓ 推荐(= T1)** |
| C 把 IntentRouter 也并回 manus.py | 两个模块全部并回 | manus 文件数最少 | **反对**:IntentRouter 不是薄——它是每轮 process_input 单次调用契约 + 让权 + enhanced_query 清洗的唯一承载点,并回去等于把刚从 think() 拆出来的 LLM-first 纪律又塞回 794 行的 think() 上下文,倒退 | ✗ |

**推荐 B,理由三点**:
1. `ToolInvocationBuilder` 是单方法、单调用点的纯构造器 → CLAUDE.md §2.2「不为单次使用场景引入抽象」的教科书案例,内联是正解。
2. 守卫函数注入是重构期为保行为等价搭的临时脚手架,该约束已随 LLM-first 了断消失(审计 §3.D)→ 平移进 IntentRouter 是"完成一半的搬运",不是新决策。
3. `IntentRouter` 独立文件有真实价值:它把"每轮恰好一次 process_input、规则识别结果只作日志参考、enhanced_query 一律丢弃"这套 LLM-first 纪律锁在一个 119 行的接缝里,是**防止规则路由复辟的护栏文档**,不该并回 think()。

## 五、方向二:System Prompt / 人设现状评估(逐段)

### 5.1 逐段评价

| prompt 段落(行号) | 定性 | 评价 |
|---|---|---|
| 身份"你是 coco…专业 AI 简历助手"(19) | 身份层,过薄 | 有名字有职能,**无性格无信念**。是"职位描述"不是"人格"。 |

| Hybrid 简历上下文说明(31-35) | 规则,好 | 清晰告知"简历已在 context,别重复 read"。保留。 |

| 诊断输出模板(172-195) | 规则(格式契约) | 保留。 |
| 回复要求 / 建议按钮规则(197-215) | 规则 + 协议 | 保留(`%%SUGGESTIONS%%` 是协议)。 |

| `OPTIMIZE_SECTION_LLM_ADDENDUM`(260-283)"缺数字不编造/禁止空话/动词开头" | 价值观(以禁令散落在任务 addendum) | 这是**当前最接近"简历审美价值观"的一段**,但它藏在 optimize-section 专用 addendum 里,不是稳定人设核心。T2 应把它的精神上提为全局价值观。 |

### 5.2 核心发现

**"一项都没落地"需修正为**:身份 + 交互层语气 + 反编造价值观**已部分隐含落地且分层正确**(安全内核在 Python、人设在 prompt 文本,互不污染);缺的是**把散落的、否定式的审美禁令,收敛成一个正向、稳定、可跨会话一致的价值观核心块**。今天实测"LLM 拒绝编造、反问用户"证明现有隐含价值观已在起效——所以 T2 是**顺势强化**,不是"从零建人设",更不该大改。

### 5.3 改进方向对比示例(不写最终版,给方向)

**现状(价值观以否定式散落三处)**:
```
（61-93 行流程里）"不要凭空创造" / "不能凭空创造"
（76 行）"严禁凭空编造占位简历"
（274-275 行 addendum）"缺数字时不编造" / "禁止空话与重复"
```

**改进方向(在开篇产品语境后,增设一个稳定核心块,把上面散落的判断收敛为正向信念)**:
```
## coco 的简历观（你的专业判断来自这些信念，不是逐条对照的清单）

- 好简历用事实说话：动词开头、可量化的结果、真实的技术细节。空泛形容词
  （"认真负责""精通各项技能"）是噪音——遇到时我会主动指出，并引导用户
  换成具体证据，而不是默默保留。
- 我绝不替用户编造没发生过的数字或经历。缺数据时我宁可反问，也不用假数据
  把简历撑好看——这是底线。
- 用户没明说的地方，我按"这样改对求职者更有利吗"来判断，而不是含糊两可、
  什么都不建议。有观点地给专业意见，是我的价值。
```
配套改动:原来 61-93、274-275 里那些禁令句可**简化引用**这个核心块(如"按 coco 简历观改写"),避免同一价值观在 prompt 里重复五遍。

**关键分寸**(必须写进 T2 交付说明):
1. 上面的价值观是**表达/判断层**;"绝不编造"作为价值观写进 prompt,但它的**强约束仍靠 Python 决策内核**(只读拦截、字段规范化),二者分层,prompt 改动不触碰任何 `.py` 决策逻辑。
2. **只价值观化"审美判断"**;HTML 格式、path、协议、诊断模板**原样保留为规则**。
3. 目标是"让边缘场景(规则没覆盖的简历怪象)也能靠价值观自处",不是"删掉规则让模型自由发挥"。

## 六、风险分级:哪些够小可直接排期,哪些必须走完整设计流程

参照 CLAUDE.md §4.2 场景流程链:

| 任务 | 场景归类 | 是否需 `/brainstorming` | 是否需 `/writing-plans` | 是否需实测 | 判定理由 |
|---|---|---|---|---|---|
| **T1 文件级收尾** | 重构 | **否** | **轻量即可**(范围已在本文锁定,可直接按 §三 T1 拆步) | **是,必须**(SSE 冒烟三条活路径) | 纯搬运/内联,无新功能、无设计歧义,行为零变化目标明确。走"重构"链:writing-plans(轻)→ simplify → code-review → 实测。**够小,可直接排期**。 |
| **T2 人设收敛** | 类"新功能"(人设设计) | **是,必须** | **是** | **是,必须(行为前后对比)** | 虽只改 prompt 文本,但 prompt 是 LLM-first 下**驱动全部行为的决策核心**。"coco 该有哪些价值观、语气到什么程度、如何不退化成过度中立/爱编造"是主观、产品向、需对齐的设计问题,且高回归风险(可能毁掉已观测到的好行为)。**必须先 brainstorming 对齐"要给什么价值观"再动**,不能直接派 agent 改 prompt。 |
| T3 evaluator | (暂缓) | 触发后必须 | 触发后必须 | 触发后必须 | 暂不排期。 |
| T4 垂直合规 | (暂缓)新功能 | 触发后必须 | 触发后必须 | 触发后必须 | 暂不排期。 |

**一句话风险判断**:**T1 可直接排期做**(小重构,锁死范围 + 实测即可);**T2 必须先走 `/brainstorming` + `/writing-plans` 完整设计对齐**——它动的是决策内核(system prompt),且是主观人设设计,不是机械改动,直接派 agent 改 prompt 风险最高。
