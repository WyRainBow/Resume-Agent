### 1.1 独立核实新发现的残留（(d) 类，今天两轮清理没扫到的）

1. **`OPTIMIZE_SECTION_LLM_ADDENDUM`（manus.py L311-334）是孤儿死代码**。全仓 grep 确认：除定义处外**零消费者**（`prompt/__init__.py`、`agent/manus.py`、`prompt_builder.py` 均未导入它）。原方案记忆里「最接近简历审美价值观的一段」现在**根本没在驱动任何行为**。但它的内容质量很高——「四要素融合（规模/手段/指标/价值）」「动词开头」「禁止 XX% 占位符、STAR 套话」「缺数字时不编造」——是核心块文案的最佳素材来源。建议：**吸收精华进核心块后删除本体**（见 D3）。
2. **manus.py L271 `SYSTEM_PROMPT.replace("<<GREETING_EXCEPTION_SECTION>>", …)` 是 no-op 死代码**。SYSTEM_PROMPT 文本里**不存在**这个占位符（grep 确认），replace 什么都没替换；`GREETING_EXCEPTION_SECTION` 的 import 也随之无效。greeting 例外实际走的是 `agent/manus.py` 里的 `GREETING_FAST_PATH_PROMPT` 独立链路。见 D4。
3. **反编造价值观在 prompt 里散落重复了至少 3 次**（L76 / L93 / L117），措辞各异（「严禁凭空编造占位简历」「不要凭空创造」「不能凭空创造」）——正是「同一价值观没有单一事实来源」的形态，收敛的直接动机。

### 1.2 分层现状复核（核实原方案的评估，结论：基本属实）

- **安全内核在代码层**：隐私脱敏走 `prompt_builder.py` 的 `mask_pii`（复用 `Intent.FULL_OPTIMIZE` 判定，今天 `1159e106` 修正）；终止兜底走 `should_auto_terminate` + `max_steps=20`（今天 `fb519ef3` 删掉强制终止后的形态）；只读/新增经历轮次约束走 `prompt_builder.py` 动态 addendum。**代码层没有混入审美判断**——分层正确。
- **人设审美在 prompt 文本层**：但呈「碎片态」——身份一句话（L19）、审美埋在技能专项（L135）、反编造重复三处、语气禁令在回复要求（L252-253）、最好的审美素材在死代码里（L311-334）。**缺的确实是统一的价值观核心块**，与原方案评估一致。

## 3. 方案对比

### 方案 A：最小收敛 —— 开头人设核心块，散落禁令删除合并

**位置**：prompt 最开头，替换现在的单句身份行（L19），核心块之后紧接「产品语境」。

**做法**：新增一个约 350-450 字的「coco 的简历价值观」区块；§2.1 表里的散落禁令**全部删除原处文本**，只在核心块里保留唯一表述。ACI 段落零改动。

**核心块文案（已定稿，D2 拍板结果——语气取更克制的"懂行的从业者"而非"简历伙伴"，避免 R7 指出的人格化盖过专业输出的风险）**：

> ## 你是谁：coco 的简历价值观
>
> 你是 coco——一个懂招聘、有审美、专业判断力强的简历助手。你所有的判断都以这四条价值观为准：
>
> 1. **真实是底线。** 简历上每个字都必须来自用户的真实经历。数据、成果、经历，原文有就用原文的，没有就开口问用户要——宁可多问一句，也绝不编一个数字、一个「张三」式占位人。缺量化数据时，把动作、技术手段、系统价值写扎实就够了：不放「XX%」占位符，不写「建议补充量化结果」这种空话。
> 2. **写出成熟从业者的味道。** 好的一条经历，是规模/场景、技术手段、量化结果、业务价值自然融进一两句话，动词开头。不写「我具备…」「本人熟悉…」的学生腔，不堆砌工具名，不写公式化的 STAR 套话（动作+成果+量化的实质表达是好的），不浮夸——读起来像一个能独当一面的人在陈述自己做成的事。
> 3. **尊重写得好的内容。** 你的任务是让简历更好，不是证明你改过它。哪里有提升空间就动哪里；本来就写得好的段落，说一句「这段已经不错」然后放过它。
> 4. **果断、说人话。** 用户说改就改，不反复要确认，也不预告动作（「您确认要这样更新吗？」「我将为您读取…」都是错误行为——直接做）。和你对话的是普通求职者：他在网页里跟你聊天，右侧实时看到简历变化，他的世界里没有目录、路径、文件——你的话里也不该有。
>
> 这四条管的是「怎么判断、怎么表达」。工具怎么调、path 怎么写、HTML 格式怎么给，以下文各节的技术规则为准——那些该精确的地方，一个字符都要精确。

**利**：
- 改动面最小、逻辑最干净：价值观单一事实来源，收敛最彻底
- 开头位置 = 身份定义位，符合「先知道自己是谁，再看技术规则」的阅读序；跨会话/跨场景一致性最好
- token 净变化约 ±0（新增 ~350 token，删除散落重复 ~200-300 token）

**弊**：
- **回归风险最高**：反编造/审美行为现在是靠 3+ 处「就地提醒」撑着的（wave2a QA 文档明确说今天的好行为来自「被规则盖住的 prompt 价值观重新生效」）。system prompt 实际运行时末尾还会追加几千 token 的简历 context + 本轮约束 addendum，开头核心块距离决策点最远，长 context 下注意力衰减是真实风险
- 技能润色、从零生成这些具体场景执行时，模型「手边」不再有风格/反编造提醒

### 方案 B（推荐）：核心块 + 就地锚点 —— 开头定义一次，原地留一行引用

**位置**：核心块同方案 A 放开头；但原禁令位置**不删空**，改为一行锚点引用。

**做法**：
- 核心块文案同方案 A 草稿
- L76 → 改为「（价值观第 1 条：绝不编造——没给信息就先问清，不要直接编）」
- L93 / L117 第 3 条 → 改为「改写必须基于 context 真实内容（价值观第 1 条）」
- L135 → 改为「风格按开头价值观第 2 条（成熟从业者表达）；自然表达如『熟悉…并能在实际项目中…』」（保留正向示例句，删除重复的否定列举）
- L252-253 两行 → 删除（已被核心块第 4 条覆盖，且「回复要求」节离核心块近、重复价值低）
- L24 「严禁文件系统话术」 → 保留一行短句（这条更接近产品防呆而非纯审美，双保险成本低）

**利**：
- **防退化最稳**：核心块负责「人设定义与跨场景一致性」，锚点负责「决策点就地触发」——两级注意力都覆盖
- 收敛目标仍达成：价值观**表述**只有一份（核心块），锚点只是指针，不会再出现三处措辞漂移
- 可渐进演进：真实 SSE 验证通过、行为稳定后，下一轮可把锚点也删掉，自然过渡到方案 A 形态

**弊**：
- token 净增约 +250~350/轮（核心块 ~350 token，锚点省不回多少）——相对整个注入后 4000+ token 的 system prompt，边际成本 <10%
- 「两级结构」需要一条维护纪律：改价值观只改核心块，锚点不允许长出自己的内容

### 方案 C：结尾价值观区 + 今日实测好行为例证

**位置**：prompt 末尾、`Current state: {context}` 之前，独立「## coco 的价值观」分区；正文引用今天真实观测到的行为作为「这是对的」例证。

**做法**：核心块四条同 A，每条附一个今日实测例证，例如：

> 1. **真实是底线。** （今天实测：用户给了一段没有量化数据的经历要求「改得更有说服力」，你反问用户要真实数据而不是编一个「提升 XX%」——这就是对的。）
> …

**利**：
- 近因效应：末尾位置紧邻简历 context 和用户消息，对当轮决策的注意力权重最高
- 例证是最强的行为锚定方式（隐式 few-shot），比抽象陈述更能复现具体好行为

**弊**：
- **例证本质是隐性模板**：写得稍不注意就退化成「遇到 X 场景必须做 Y」的场景清单——正是今天踩坑 2 的形态换了张脸（用「例证」包装的规则清单）。与架构准则的判断口诀直接冲突风险最大
- 末尾区要和简历 context、只读/新增经历动态 addendum 抢注意力，且 addendum 是运行时追加在更后面，位置优势并不稳定
- 开头身份句依旧单薄，「人设定义」与「行为提示」割裂
- 例证会随版本演进过时（引用的是 2026-07-11 的行为快照），维护负担最重

### 方案对比表

| 维度 | A：开头核心块（删净） | B：核心块+就地锚点 | C：结尾块+例证 |
|---|---|---|---|
| 收敛彻底度 | 最高 | 高（表述唯一，锚点是指针） | 中（例证是新增内容形态） |
| 回归风险 | **高**（就地提醒全拔） | **低** | 中-高（例证易变规则清单） |
| token 成本 | ±0 | +250~350/轮 | +400~500/轮 |
| 与今日方法论一致性 | 一致 | 一致 | **有冲突风险**（隐性 few-shot 清单化） |
| 可演进性 | 终态 | 可演进为 A | 需持续维护例证 |
| 维护纪律要求 | 低 | 中（改价值观只改核心块） | 高（例证保鲜） |

## 4. 推荐方案：B（核心块 + 就地锚点）

理由：

1. **这次改动的第一目标不是「prompt 更漂亮」，是「不退化」。** wave2a QA 文档（`2026-07-11-agent-wave2a-summary-qa.md` §五）明确记录：今天观测到的「拒绝编造、主动追问」好行为，根源是散落在 prompt 里的既有价值观「重新生效」——这些价值观现在就长在 L76/L93/L117/L135 那些位置上。方案 A 一次性把它们全部搬到几千 token 之外的开头，等于拿**已被真实 SSE 验证过的行为**去赌注意力机制，赌输的代价（反编造退化）恰好是本产品最不能接受的失败模式。
2. **B 完成了收敛的实质目标**：价值观的**表述**从 3+ 处措辞各异收敛为核心块单一来源，跨会话一致性、可维护性都达标；锚点只是一行指针，不构成第二来源。
3. **B 是通往 A 的安全路径**：验证通过、跑过一段真实使用后，删锚点只是一次低风险的减法；反过来如果 A 上线后发现退化，再往回加就地提醒就是一次带着事故压力的紧急修复。
4. C 的例证思路单独不推荐，但其「近因位置」洞察可以当 B 的备用手段：**如果**验证阶段发现 B 的开头核心块在长简历 context 下有明显注意力衰减，可把核心块整体挪到末尾（不带例证），作为 B 的降级预案，而不是默认设计。
