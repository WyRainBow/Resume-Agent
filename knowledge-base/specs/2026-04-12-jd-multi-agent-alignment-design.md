# Spec: JD 驱动的多 Agent 简历对齐与学习路径

> 最后更新：2026-04-12

## 1. 目标

将 Resume-Agent 从“通用简历助手”升级为“面向目标岗位的求职助手”。

V1 目标是围绕单个目标岗位 JD，完成以下闭环：

1. 获取并保存 JD
2. 用当前 AI 助手会话中的简历数据进行岗位匹配分析
3. 生成可逐模块确认的简历改写 Patch
4. 输出岗位强绑定的技术栈缺口分析
5. 输出分阶段学习路径与项目产出建议

## 2. V1 范围

### 2.1 输入

- 单份简历 + 单个 JD，一次分析
- 分析对象为“当前 AI 助手会话中已加载的简历数据”
- JD 输入支持：
  - 粘贴 JD 文本
  - 输入公开招聘链接抓取 JD

### 2.2 JD 管理

- 每个用户可保存多个 JD
- 每个用户可设置一个默认 JD
- JD 为用户私有数据，仅本人可见
- 默认 JD 可长期复用

### 2.3 输出

分析结果固定为三个标签页：

- 报告
- Patch
- 学习路径

约束如下：

- 报告：匹配度摘要、核心缺口、优先修改项、技术栈建议
- Patch：仅包含简历改写 Patch，不混入学习建议
- 学习路径：仅包含阶段目标、学习主题、项目建议、成果表达

## 3. 入口与交互

### 3.1 功能入口

- V1 入口只放在 AI 助手页
- 不在简历编辑页新增固定 JD 面板
- AI 助手页提供“岗位对齐 / JD 分析”侧边抽屉

### 3.2 抽屉内容

抽屉至少包含以下区域：

- 默认 JD 区
- 已保存 JD 列表
- 新建 JD
- JD 输入方式切换：链接 / 文本
- 运行分析按钮
- 阶段进度
- 结果标签页：报告 / Patch / 学习路径

### 3.3 意图触发

当用户在 AI 助手输入中表达以下意图时，应自动展开 JD 分析抽屉：

- 岗位分析
- JD 匹配
- 岗位对齐
- 针对某岗位优化简历
- 分析招聘链接
- 分析岗位描述

触发后的交互要求：

- 不直接沿用普通聊天流完成整套 JD 分析
- 优先引导用户在抽屉中完成 JD 输入
- 若输入中包含公开招聘链接，可预填到“岗位链接”输入框
- 若输入中包含明显的岗位文本描述，可预填到“岗位文本”输入框
- 分析必须由用户显式点击“运行分析”触发

### 3.4 前置条件

- 若当前 AI 助手会话中没有已加载简历，则禁止运行分析
- 系统必须明确提示用户先加载或选择简历
- 不允许对空简历上下文执行 JD 分析

## 4. 分析流程

1. 用户主动打开，或通过岗位分析意图触发打开 JD 分析抽屉
2. 用户选择默认 JD、已保存 JD，或新建一个 JD
3. 若输入为链接，系统抓取并结构化 JD
4. 系统以当前 AI 助手会话中的简历数据作为分析输入
5. 主 Agent 编排多个专用子 Agent 完成分析
6. 前端按阶段展示进度
7. 分析完成后，在抽屉内展示三个标签页结果
8. 用户可在 Patch 标签页逐模块确认是否应用修改
9. 应用 Patch 仅更新前端简历编辑态，不自动保存数据库
10. 用户仍需通过现有保存动作手动持久化简历

## 5. Patch 确认机制

### 5.1 粒度

- V1 不支持“一键应用全部 Patch”
- Patch 必须按模块确认

模块示例：

- 基本信息
- 个人总结
- 工作经历
- 项目经历
- 技能
- 教育经历

### 5.2 应用规则

- 每个模块可独立应用或跳过
- 应用后只更新当前前端工作区中的简历状态
- 不自动保存到数据库
- 用户必须手动保存

## 6. 多 Agent 协作

### 6.1 主 Agent

- `JDAlignmentOrchestrator`
- 负责统一编排、汇总结果、输出统一结构

### 6.2 子 Agent

- `JDParserAgent`
  - 抓取招聘链接内容
  - 提取并结构化 JD

- `JDRequirementAgent`
  - 提炼岗位职责、必备技能、加分项、技术方向

- `ResumeMatchAgent`
  - 用当前简历与结构化 JD 做差距分析
  - 输出匹配度、核心缺口、优先修改项

- `ResumePatchAgent`
  - 基于差距生成模块化简历 Patch
  - 输出供前端逐模块确认的 Patch 批次

- `LearningPathAgent`
  - 根据岗位方向和当前差距输出学习路径
  - 每阶段输出项目建议与可写入简历的成果表达

## 7. 数据持久化

### 7.1 JD 记录

每条 JD 至少包含：

- 岗位标题
- 公司名
- 来源链接
- 原始 JD 文本
- 结构化提取结果
- 创建人
- 抓取时间
- 最近使用时间
- 是否为默认 JD

### 7.2 分析结果

JD 与分析结果是一对多关系。

每次分析都应保存为独立记录，至少包含：

- 关联用户
- 关联简历
- 关联 JD
- 分析时间
- 匹配度摘要
- 核心缺口
- 技术栈建议
- 学习路径
- 模块化 Patch 批次

## 8. 结果要求

### 8.1 报告页

必须包含：

- 总体匹配度 / 匹配评分
- 核心缺口摘要
- 优先修改项
- 当前必补技术栈
- 未来储备技术栈

要求：

- 所有建议必须强绑定目标岗位
- 不允许输出泛化的通用建议

### 8.2 学习路径页

每个阶段至少包含：

- 阶段目标
- 重点学习主题
- 建议完成的项目 / 作品
- 可写入简历的成果表达

要求：

- 必须根据岗位差距动态生成
- V1 不输出课程、文章、视频、开源仓库等外部资源链接

## 9. 阶段进度

前端必须展示明确阶段进度，而不是只展示最终结果。

建议阶段固定为：

1. 获取 JD
2. 结构化 JD
3. 提炼岗位要求
4. 分析简历差距
5. 生成 Patch
6. 生成学习路径

约束：

- V1 只展示阶段状态
- 不暴露子 Agent 的完整内部推理

## 10. 错误处理

- JD 链接抓取失败时必须明确报错
- 必须提示用户改为粘贴 JD 文本
- 不允许静默降级为“抓取失败但继续分析空内容”
- 没有简历上下文时必须明确阻止分析，而不是返回空报告

## 11. 项目落点

本次需求基于现有系统扩展，不另起新系统。

主要落点：

- AI 助手页 JD 分析抽屉
- AI 助手输入区的岗位分析意图触发逻辑
- JD 数据模型与持久化接口
- JD 分析结果模型与持久化接口
- 现有 Agent 编排链路扩展为 JD 驱动工作流
- 现有 Patch 机制升级为模块级批次确认

## 12. 不在 V1 范围内

- 在简历编辑页增加固定 JD 面板
- 同时分析多份简历
- 同时分析多个 JD 并横向比较
- 登录后页面或需要权限页面的 JD 抓取
- 图片 JD / PDF JD 的专门解析流程
- 自动保存 Patch 到数据库
- 外部课程、文章、视频、开源资源推荐
