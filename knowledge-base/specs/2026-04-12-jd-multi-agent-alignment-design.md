# Spec: JD 驱动的多 Agent 简历对齐与学习路径

> 最后更新：2026-04-12

## 1. 背景与目的

当前项目已经具备简历编辑、Agent 对话、简历分析、Patch 应用等基础能力，但还缺少围绕“目标岗位 JD”展开的一体化工作流。

本次改动的目标是把现有 Resume-Agent 从“通用简历助理”升级为“面向目标岗位的求职助理”，让用户可以围绕特定 JD 完成以下闭环：

1. 获取并保存目标岗位 JD
2. 将当前正在编辑的简历与该 JD 做差距分析
3. 生成可确认应用的简历改写建议
4. 输出岗位强绑定的技术栈缺口分析
5. 生成按阶段推进的学习路径与项目产出建议

## 2. V1 范围

### 2.1 输入范围

- 单份简历 + 单个 JD 进行一次分析
- 分析对象默认是“当前工作区正在编辑的简历数据”
- JD 输入支持两种方式：
  - 粘贴 JD 文本
  - 输入公开招聘链接抓取 JD

### 2.2 JD 管理范围

- 每个用户可以保存多个 JD 记录
- 每个用户可以设置一个默认 JD
- 默认 JD 链接可长期保存，后续分析可以直接复用
- JD 数据为用户私有数据，仅该用户可见

### 2.3 输出范围

分析结果按三个标签页展示：

- 报告
- Patch
- 学习路径

其中：

- 报告：包含匹配度摘要、核心缺口、优先修改项、技术栈建议
- Patch：只包含“简历改写 patch”，不自动写入学习路径或技术栈建议
- 学习路径：输出分阶段学习计划、项目建议、可写入简历的成果表达

## 3. 关键用户流程

### 3.1 工作区入口

- 在工作区左侧加入固定的“岗位对齐 / JD 分析”面板
- 面板至少包含以下功能区：
  - 默认 JD 区
  - 已保存 JD 列表
  - 新建 JD（文本 / 链接）
  - 运行分析按钮

### 3.2 分析流程

1. 用户在工作区打开左侧“岗位对齐 / JD 分析”面板
2. 用户选择已有 JD、默认 JD，或新建一个 JD
3. 如果是链接方式，系统抓取并结构化 JD
4. 系统以当前工作区正在编辑的简历数据作为分析输入
5. 主 Agent 编排多个专用子 Agent 完成分析
6. 前端按阶段展示进度
7. 分析完成后展示三个标签页结果
8. 用户可在 Patch 标签页按模块确认是否应用修改
9. Patch 仅更新前端编辑态，仍由用户手动保存简历

## 4. Patch 确认机制

### 4.1 确认粒度

- V1 不采用“整批一次确认全部应用”
- Patch 必须改为“按模块逐个确认是否修改”

模块示例：

- 基本信息
- 个人总结
- 工作经历
- 项目经历
- 技能
- 教育经历

### 4.2 应用规则

- 每个模块可以独立确认应用或跳过
- 应用后只更新当前前端工作区中的简历状态
- 不自动保存到数据库
- 用户仍需通过现有保存动作手动持久化简历

## 5. 多 Agent 协作设计

### 5.1 主 Agent

- `JDAlignmentOrchestrator`
- 负责串联整个流程、汇总结果、对外输出统一结构

### 5.2 专用子 Agent

- `JDParserAgent`
  - 抓取招聘链接内容
  - 提取并结构化 JD
- `JDRequirementAgent`
  - 提炼岗位职责、必备技能、加分项、技术方向
- `ResumeMatchAgent`
  - 用当前工作区简历与结构化 JD 做差距分析
  - 输出匹配度、核心缺口、优先修改项
- `ResumePatchAgent`
  - 基于匹配差距生成模块化简历 patch
  - 输出可供前端逐模块确认的 patch 批次
- `LearningPathAgent`
  - 根据岗位方向和当前差距输出动态学习路径
  - 每阶段给出项目建议与可写入简历的成果表达

## 6. 数据持久化设计

### 6.1 JD 记录

每条 JD 至少包含：

- 岗位标题
- 公司名
- 来源链接
- 原始 JD 文本
- 结构化提取结果
- 创建人
- 抓取时间
- 最近使用时间
- 是否为默认 JD

### 6.2 分析结果

JD 与分析结果是一对多关系。

每次分析都应形成一条独立结果记录，至少包含：

- 关联用户
- 关联简历
- 关联 JD
- 分析时间
- 匹配度摘要
- 核心缺口
- 技术栈建议（当前必补 / 未来储备）
- 学习路径
- 模块化 patch 批次

## 7. 报告页要求

报告页必须包含：

- 总体匹配度 / 匹配评分
- 核心缺口摘要
- 优先修改项
- 当前必补技术栈
- 未来储备技术栈

所有建议必须强绑定目标岗位方向，不能输出泛化的通用建议。

## 8. 学习路径要求

学习路径不采用固定模板，必须根据岗位差距动态生成。

每个阶段至少包含：

- 阶段目标
- 重点学习主题
- 建议完成的项目 / 作品
- 未来可写入简历的成果表达

V1 不要求输出外部资源链接、课程链接、开源仓库链接。

## 9. 阶段进度反馈

前端需要展示明确的阶段进度，而不是只给最终结果。

建议阶段包括：

1. 获取 JD
2. 结构化 JD
3. 提炼岗位要求
4. 分析简历差距
5. 生成 Patch
6. 生成学习路径

V1 只展示阶段进度，不暴露子 Agent 的完整内部推理。

## 10. 错误处理

- JD 链接抓取失败时必须明确报错
- 必须提示用户改为粘贴 JD 文本
- 不允许静默降级为“抓取失败但继续分析空内容”

## 11. 当前项目中的落点

本次需求默认基于当前项目已有能力扩展，不另起一套系统。

主要落点包括：

- 工作区左侧固定面板扩展
- JD 数据模型与持久化接口
- JD 分析结果模型与持久化接口
- 现有 Agent 编排链路扩展为 JD 驱动工作流
- 现有 Patch 机制从“单条”升级为“模块级批次确认”

## 12. 明确不在 V1 范围内

- 同时分析多份简历
- 同时分析多个 JD 并横向比较
- 登录后页面或需要权限页面的 JD 抓取
- 图片 JD / PDF JD 的专门解析流程
- 自动保存 Patch 到数据库
- 外部课程、文章、视频、开源资源推荐
