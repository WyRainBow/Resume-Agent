# HTML 模板照片接入方案(排查 + 方案,未实施)

> 状态:排查完成,方案待拍板 ｜ 分支:feature/template-market ｜ 日期:2026-07-09

## 一、根因

`basic.photo` 数据是完整的(字段定义 + 上传逻辑齐全),但只有 **LaTeX 渲染链路**接了照片,**HTML 模板链路(Builder 系统,7 套模板)从数据转换到组件渲染到导出,四个环节全部空白**——不是有 bug 不显示,是压根没接线。

## 二、现状排查

### 2.1 数据源

`basic.photo` 定义在 `frontend/src/pages/Workspace/v2/types/index.ts:19-33`(`BasicInfo` 接口):

```ts
photo?: string           // URL 字符串,不是 base64/File
photoOffsetX?: number
photoOffsetY?: number
photoWidthCm?: number
photoHeightCm?: number
```

上传逻辑:`EditPanel/BasicPanel.tsx:58-67` → `services/photoService.ts:9-30` → POST `/api/photos/upload` → 返回 `{ url, key }` → 写回 `basic.photo`。全程是一个后端可访问的图片 URL,不涉及 base64/File 对象。

### 2.2 LaTeX 链路怎么用(作为接入参照)

- **前端**:`convertToBackend.ts:11-15` + `:107-111` 原样透传 5 个字段到后端,无转换。
- **后端**(`backend/latex_generator.py`):
  - `_download_user_photo_to_dir`(`:181-207`)把 `photo_url` **下载到临时目录** `logos/photo.<ext>`(仅接受 http/https)。
  - `:416-427` 用 `\includegraphics[width=...cm,height=...cm,keepaspectratio]{photo}` + `\raisebox`/`\makebox[\textwidth][r]` 叠加到右上角,受 offsetX/offsetY/width/height 四个参数控制。
  - `:529-544` 编译前下载图片;**下载失败会自动把含 `{photo}` 的行剔除**,优雅降级,不让编译因缺图失败。

LaTeX 是"下载到本地文件 + 文件名引用",不是直接引用远程 URL——这是编译期渲染的特殊要求,HTML 链路（浏览器直接渲染）不需要这一步，可以直接用 `<img src={url}>`。

### 2.3 HTML 模板链路的四处空白

| 环节 | 文件 | 现状 |
|---|---|---|
| 类型定义 | `Builder/types.ts:7-16`(`PersonalInfo`) | 无 photo/avatar 字段占位 |
| 数据转换 | `Builder/adapter.ts` `toPersonalInfo`(`:50-63`)、`toBuilderResumeData`(`:186-199`) | 未映射 photo;`:202` 注释提到的"照片不丢"只是存储层原样保留未知字段,**不等于渲染层能拿到** |
| 组件渲染 | `Builder/templates/` 7 套模板(Clean/ModernTwoColumn/SingleColumn/TwoColumn/Latex/Vivid/Modern)+ `ResumeRenderer.tsx` | 对 `<img>`/`avatar`/`photo`/`logo` 全部零命中,无渲染逻辑也无布局占位 |
| 缩略图 | `Builder/components/TemplateSelector.tsx:54` 起的 `TemplateThumbnail` | 纯 CSS/SVG 示意图,不展示真实照片,也无照片相关逻辑 |

### 2.4 导出链路(html2pdf)

`Workspace/v2/index.tsx:247-267` `handleDownloadHtmlPDF` 已配置 `html2canvas: { useCORS: true }`(`:261`),说明团队对跨域抓图有基本预案,但**目前代码里没有任何 `<img>` 标签**,这个配置从未被 photo 场景验证过。接入后必须验证:
1. `/api/photos/upload` 返回的图片 URL,后端是否带 `Access-Control-Allow-Origin` 响应头;
2. 若不带,html2canvas 会导出黑块/空白,而不是报错——容易被漏测。

### 2.5 可参考先例(有局限,不能直接照搬)

`Workspace/v2/utils/generateHTML.ts:45-56`(学校 Logo)和 `:72-90`(公司 Logo)有拼 `<img src="${url}">` 的先例,但:
- Logo 是**预置公共素材**(固定域名、无鉴权),`basic.photo` 是**用户私有上传图片**,跨域/权限模型不同,不能假设同样能被 html2canvas 无障碍抓取;
- `generateHTML.ts` 经排查**当前无任何调用方**,疑似死代码,其写法仅可作 HTML 结构参考,不能确认在真实导出链路里验证过。

## 三、接入方案

### 3.1 类型 + 数据转换(基础,风险低,应先做)

- `Builder/types.ts` 的 `PersonalInfo` 加 `photo?: string`。是否需要 offset/size 四个字段,见 3.3 的开放问题。
- `Builder/adapter.ts` 的 `toPersonalInfo` 补一行 `photo: data.basic?.photo`,`toBuilderResumeData` 同步透传。

这一步不涉及渲染,不影响现有 7 套模板的任何现状,可独立验证(读取 + console.log 确认字段能传到 `BuilderResumeData`)。

### 3.2 组件渲染(核心工作量,风险中)

需要在 `ResumeRenderer.tsx` 分发的每套模板文件里加 `<img src={personalInfo.photo}>`,但**不是简单加一行就行**——7 套模板当前完全没给照片留布局位:

- **单栏类模板**(SingleColumn/Clean/Vivid)通常头部是纯文字(姓名+联系方式横排),加照片要么改成"文字+右侧头像"两栏头部,要么头像浮在角落——这是**布局改动**,不是插入。
- **双栏类模板**(TwoColumn/ModernTwoColumn)本身有侧栏,头像可以放侧栏顶部,改动相对小。
- **Latex 风格模板**(ResumeLatex.tsx)理论上最适合复用 LaTeX 链路已有的 offsetX/offsetY/width/height 四个参数做同款右上角叠加效果,一致性最好。

**建议不是"一次性接 7 套"**,而是分批,见 3.4。

### 3.3 需要产品拍板的开放问题

在写代码前必须先定,否则会来回返工:

1. **是否所有模板都要支持照片?** 简历模板行业惯例里,部分极简风格(比如纯文字 Clean 风)刻意不放照片。可能不是"7 套全接",而是"选定几套接"。
2. **照片形状**:圆形(现代简历模板常见)、方形直角(neo-brutalism 一致)、还是复用 LaTeX 的矩形裁切?这直接决定 CSS 写法(`rounded-full` vs `rounded-none`)。
3. **是否复用 LaTeX 的 offsetX/offsetY/width/height 四个精确控制参数?** HTML/CSS 布局比 LaTeX 绝对定位灵活得多(flexbox/grid 能让每个模板自己控制头像位置),**建议简化**:HTML 侧只保留一个"是否显示照片"开关 + 可能的"尺寸"档位(小/中/大),不做像素级 offset 拖拽——这能大幅减少 FormattingControls 要新增的控件,也避免用户在 LaTeX 和 HTML 模板之间切换时被两套不同的调整逻辑搞晕。这一条对工作量影响最大,建议优先拍板。
4. **FormattingControls 要不要加照片开关?** 还是像 LaTeX 链路一样,"有 photo 字段就自动显示,没有就不显示"(零配置,跟现状行为一致,复杂度最低)。

### 3.4 建议的分阶段实施顺序

1. **Phase 1**(基础,~1 天):补 3.1 的类型 + adapter 映射,不接渲染,先确保数据能流通到 `BuilderResumeData`。
2. **Phase 2**(试点验证,1 套模板):选 1 套结构最简单的模板(建议 ModernTwoColumn,侧栏天然有头像位)接入渲染 + 验证 html2canvas 导出(2.4 的 CORS 风险在这一步暴露)。CORS 若有问题,先在这一步解决(联系后端确认 `/api/photos/upload` 响应头,或前端加 `crossOrigin="anonymous"`),不要等 7 套全接完再发现。
3. **Phase 3**(推广,剩余 6 套):试点验证通过后,按 3.3 定好的形状/尺寸规则,逐套补齐其余模板,每套都要过一遍真实导出验证(不能只看屏幕预览,html2canvas 渲染结果可能和浏览器预览不一致)。
4. **Phase 4**(收尾):`TemplateThumbnail` 缩略图是否要体现"这套模板支持照片"的视觉提示,产品决定后再做,优先级最低。

### 3.5 风险汇总

| 风险 | 等级 | 说明 |
|---|---|---|
| CORS 导出黑块 | 中 | html2canvas 对跨域图片的行为是静默失败(黑块),不是报错,容易被漏测,建议 Phase 2 优先验证 |
| 7 套模板逐一改布局 | 中 | 不是加一行 `<img>` 能解决,部分模板头部要重新排版才有照片位,工作量取决于 3.3 的范围决定 |
| 与 LaTeX 链路行为不一致 | 低 | 若 3.3 决定简化参数(不做 offset 拖拽),用户会发现"同一份简历,LaTeX 模板能精确调照片位置,HTML 模板不能"——需要产品认可这个体验差异是可接受的 |
| generateHTML.ts 先例不可靠 | 低 | 该文件无调用方,仅供 HTML 结构参考,不要直接复用其导出验证结论 |

## 3.6 实施记录(2026-07-09,首个原型 + 复用/去重)

已落地(未提交):

**后端 · COS 存储去重(`backend/routes/photos.py`)**
- 上传文件名从随机 `uuid` 改为**内容 SHA-256 哈希**(`_build_photo_key`)——同一张图恒映射同一 COS 对象,天然去重,不再累积重复对象。
- 上传前 `object_exists` 探测,已存在则跳过 PUT,响应新增 `reused` 布尔(附加字段,不破坏接口)。
- 新增 `GET /api/photos/list`——列出当前用户 `users/<account>/photos/` 前缀下的照片,供前端复用选择。
- ⚠️ 运行中的后端(:9100,无 `--reload`)需**重启**才能加载上述改动;真实 COS 往返(去重命中、列表)需登录态 + COS 凭证实测,受"不能代填密码"限制,由用户验证。

**前端 · 数据链路 + 一个模板渲染 + 复用选择器**
- `Builder/types.ts` `PersonalInfo` 加 `photo?: string`;`Builder/adapter.ts` `toPersonalInfo` 映射 `basic.photo`——数据打通到 `BuilderResumeData`。
- **仅 `ResumeModernTwoColumn` 接了渲染**(试点):头部改左文右图 flex,`<img>` 固定 76×95 方形、模板强调色描边;有 `photo` 才显示(零配置,同 LaTeX 行为)。**其余 6 套模板未接**。
- `BasicPanel` 加"从已上传照片中选择"选择器(`listUserPhotos` → 缩略图网格 → 点选复用),避免重复上传。

**本原型刻意简化(对应 §3.3 开放问题的临时取舍,待正式定案)**
- 只接 1 套模板(ModernTwoColumn),非 7 套。
- 照片形状/尺寸写死(76×95 方形),**未复用 LaTeX 的 offsetX/Y/width/height 精确参数**(§3.3 问题 3 倾向简化,此原型即按简化走)。
- 未设 FormattingControls 开关,走"有 photo 就显示"。
- **导出 CORS 未验证**(§2.4 风险):模板 `<img>` 未加 `crossOrigin`,屏幕预览可显示,但 html2canvas 导出跨域照片可能黑块,需 §3.4 单独验证。

## 四、下一步

方案已具备实施条件,但 **3.3 的四个开放问题需要先拍板**(尤其是"是否所有模板都要照片"和"要不要复用 LaTeX 的精确 offset 参数"),否则 Phase 2/3 的工作量无法准确估计。建议按 CLAUDE.md 流程,这类涉及多模板布局改动的功能应先过 `/superpowers:brainstorming` 定案,再排 `/superpowers:writing-plans` 详细计划。
