# 邮箱发送简历功能 —— 设计方案

- **日期**:2026-07-08
- **状态**:设计讨论中,**未写代码**(用户明确"先不做后端,先设计")
- **需求来源**:用户(运营/做简历优化服务)想在系统里点一个按钮,把优化好的简历(PDF)通过邮箱发给客户/招聘方;发件人是自己。当前**是否做成多用户暂不确定**,设计需为"先单发件人、后可扩多用户"留余地。
- **关联调研**:[[2026-07-08-resume-matcher-outreach-mail-调研]](RM 的 OUTREACH MAIL 是"生成文案"不是"真发信",本功能是真发信,二者互补)

---

## 一、需求澄清(讨论结论)

- **要做的是"真发邮件 + 带简历 PDF 附件"**,不是 RM 那种只生成文案让用户自己复制。
- **发件人 = 用户本人邮箱**(先是运营自己一个 Gmail)。
- **收件人**:用户手填一个/多个邮箱(客户、招聘方)。
- **附件**:那份简历的 PDF。
- **不做**:多用户各自授权 Gmail(受限 scope + Google 年度安全评估,成本过高,见下"路 A 否决")。

---

## 二、为什么不走"每个用户授权自己 Gmail"(路 A 否决记录)

- 从用户自己 Gmail 发,技术上是 Gmail API + OAuth 加 `gmail.send` scope。
- `gmail.send` 是 Google 的**受限 scope(restricted)**:面向公众的产品必须做 OAuth 验证 + **每年一次第三方安全评估(CASA)**,费用约 $1k~上万美金/年。
- 不验证只能停"测试"状态:≤100 用户、refresh token 7 天过期,生产不可用。
- 且只覆盖 Gmail 用户(我们用户用 QQ/163/Outlook 的发不了)。
- **结论**:产品级"每人授权自己 Gmail"不划算,否决。个人/单发件人场景用更简单的 App Password + SMTP。

---

## 三、简历数据从哪来

**从我们自己的简历系统来**,复用现有 PDF 渲染链路,不另造数据源:

```
简历(ResumeData) → convertToBackendFormat → /api/pdf/render/stream → json_to_latex → XeLaTeX → PDF bytes
                                                                                              ↓
                                                                          作为邮件附件发出(本功能新增)
```

- 用户在系统里选一份简历 → 后端用**已有的 PDF 渲染**生成 PDF 字节 → 邮件发送器把它当附件发出去。
- 不新增简历数据模型,不碰 PDF 渲染逻辑,只是"渲染完的 PDF 多一个出口:发邮件"。

---

## 四、核心设计原则:传输层配置驱动(保证本地/线上都能用)

> 这是本设计的重点,直接回答"怎么保证代码在本地和服务器都能发出去"。

### 4.1 为什么"本地能、线上不能"是真实风险
我们服务器是**腾讯云、中国大陆**(`106.53.113.137`)。让本地能跑、线上跑不了的,**不是代码,是网络链路**:

1. **GFW(最致命)**:大陆服务器连 `smtp.gmail.com` 不稳定(Google 被墙/限速)。本地在墙外/挂梯子能发,同一份代码上大陆服务器大概率超时。**代码绕不过。**
2. **云厂商封 SMTP 出站端口**:腾讯云等默认封 25 出站,有时限 465/587。
3. **`.env` 未同步**:`.env` 是 gitignored,不随代码推送;服务器不手动补配置就"未配置"。

### 4.2 保证跨环境可用的三条设计原则
不是"保证 Gmail 一定能发",而是"**代码不写死链路,每个环境走各自能通的路**":

1. **传输层完全配置驱动,代码不绑定 Gmail**
   抽象一个"邮件发送器"接口(如 `EmailSender.send(to, subject, body, attachments)`);SMTP host/port/user/pass 或 API key **全部从 env 读**。本地 env 填 Gmail,服务器 env 填大陆可达通道 —— **同一份代码,换配置切换**,不改逻辑。
2. **启动自检 + 测试发送接口**
   服务启动时(或提供一个「测试发送」端点)试连一次配置好的通道,连不上就在日志明确报错。**部署完立刻知道通不通**,而不是等用户点发送才暴露。
3. **env 校验 + 部署清单,失败要吵不要静默**
   启动检查发件相关 env 键是否齐全,缺了明确报错;部署文档写死"服务器 `.env` 必须补这几个键"。发送失败要把错误透出,不吞异常走假成功。

### 4.3 推荐的"双通道"落地
| 环境 | 通道 | 理由 |
|---|---|---|
| 本地开发 / 个人用 | Gmail App Password + SMTP(`smtp.gmail.com:587`) | 能真从自己 Gmail 发,墙外/梯子可达 |
| 线上服务器 | **腾讯云邮件推送(SES)** 或 QQ/163 邮箱 SMTP | 同云内网可达 / 墙内可达,走 443 或墙内端口,不受 GFW 和端口封锁影响,几乎不可能失败 |
| 代码 | **同一套**,靠 §4.2 配置驱动无缝切换 | |

- 若坚持"线上也必须从 Gmail 发出":需加墙外中转(代理/中转服务器转发 SMTP),多一个故障点,**不建议**;用 `Reply-To 设成用户 Gmail` 通常够用(收件人回信回到你 Gmail)。

## 五、密钥/配置放哪、安不安全

用户问:密钥放 `.env` 行不行,安全吗?

- **单发件人阶段:放 `.env` 是合适的**,前提:
  - 用的是 **App Password(应用专用密码),不是 Gmail 主密码** —— 它是 Google 专门给第三方客户端的、可随时在 Google 账号里单独吊销、权限仅限发信,泄露影响可控。
  - `.env` **已 gitignored,绝不进 Git**;不在日志/报错里打印完整密钥(复用项目现有 config 的"只显示前后几位"惯例)。
  - 服务器 `.env` 权限收紧(`chmod 600`)。
- **env 键(拟定)**:
  ```
  MAIL_TRANSPORT=smtp            # smtp | tencent_ses(决定走哪个发送器)
  MAIL_SMTP_HOST=smtp.gmail.com
  MAIL_SMTP_PORT=587
  MAIL_SMTP_USER=you@gmail.com
  MAIL_SMTP_PASSWORD=<App Password 16 位>
  MAIL_FROM_NAME=你的名字 / 品牌
  MAIL_REPLY_TO=you@gmail.com    # 可选,收件人回信落到这里
  ```
- **未来做多用户时**(每个用户填自己的发件邮箱)**才需要升级**:密钥不能再放全局 `.env`,要**每用户一份、加密入库**(参考 RM 用 Fernet 对 API key 做静态加密存 SQLite 的做法)。当前单发件人不必上这套,但接口设计上把"取哪份发件配置"留成一个可替换的点,将来从"读 env"换成"读用户加密配置"即可,不推翻结构。

## 七、待定 / 讨论点

1. **多用户与否**:先单发件人(运营自己),接口预留可扩;真要多用户再上"每用户加密配置 + 各自 SMTP"。
2. **线上发件身份**:接受"腾讯云 SES + 自有域名发,Reply-To 用户邮箱"?还是必须 Gmail 本尊(需墙外中转)?—— 倾向前者。
3. **域名**:换 `.com`(如 cocoresume.com)后配好 SPF/DKIM/DMARC,自有域名发信更不易进垃圾箱(见域名讨论)。
4. **正文/文案**:是否复用 Coco 生成一段外联话术(见 RM outreach 调研),还是用户纯手写。
5. **发送记录/频控**:是否要记发送历史、防滥用限频。

## 八、Gmail / QQ 邮箱都接得了吗(2026-07-08 补充)

用户问:"能不能接入 Google 邮箱、QQ 邮箱也可以,一键把优化的简历发给对面"——答案:**能,且本设计的 §4.3 双通道方案本来就是按"Gmail + QQ/163 都走 SMTP"设计的**,不需要新方案,细化如下:

### 8.1 两者都是标准 SMTP,同一套 `SmtpSender` 通吃

不需要为 Gmail、QQ 各写一个实现——都是标准 SMTP 协议,`SmtpSender` 一个类,区别只是 env 里的 host/port/账号/密码不同:

| 邮箱 | SMTP Host | 端口 | 认证方式 |
|---|---|---|---|
| Gmail | `smtp.gmail.com` | 587(STARTTLS)/ 465(SSL) | **App Password**(16 位应用专用密码,需账号开二步验证后在 Google 账号设置里生成,不是登录密码) |
| QQ 邮箱 | `smtp.qq.com` | 587 / 465(SSL) | **授权码**(QQ 邮箱「设置→账户→POP3/IMAP/SMTP」里开启服务后生成的 16 位码,同样不是 QQ 密码) |
| 163 邮箱(备选) | `smtp.163.com` | 465 / 994(SSL) | 授权码,同 QQ 邮箱逻辑 |

三者共同点:都不能直接拿"登录密码"当 SMTP 密码,都要求先在邮箱后台开一个"独立于登录密码"的专用凭证——这正好和 §五"密钥放 `.env`,用的是可单独吊销的专用凭证"的安全前提天然吻合,不用改设计。

### 8.2 一键发送流程 = 已设计的 §六后端结构,不用新增

用户描述的"一键把优化的简历发给对面",正是 §六里已经拟好的:

```
用户在前端点「发送到邮箱」→ 填收件人(对方邮箱)+ 主题/正文
  → POST /api/mail/send-resume
  → 后端:取当前简历 → 走已有 PDF 渲染链路(§三)生成 PDF bytes
  → SmtpSender.send(to=对方邮箱, attachments=[PDF]) —— 用 env 里配置好的发件邮箱(Gmail 或 QQ 皆可,取决于部署方选哪个)
  → 对方邮箱收到一封带简历 PDF 附件的邮件
```

不需要额外设计,§三到§六已经覆盖完整链路,唯一要做的是**运营/部署方自己决定用 Gmail 还是 QQ 当发件邮箱**,二选一或都配(比如本地开发用 Gmail 测,线上正式用 QQ/163,因为大陆服务器连 QQ/163 比连 Gmail 稳定,呼应 §四"线上优先选墙内可达邮箱服务商"的结论)。

### 8.3 哪个更适合线上(大陆服务器)发件

- **QQ / 163 邮箱**:腾讯/网易国内服务器,SMTP 在大陆网络环境下**稳定可达**,不受 GFW 影响,**推荐线上用这个**。
- **Gmail**:大陆服务器连 `smtp.gmail.com` 有被墙风险(见 §4.1),**只适合本地开发测试**,不建议作为线上唯一通道。
- 结论和 §4.3 表格一致:线上优先 QQ/163(或腾讯云 SES),Gmail 留给本地开发/测试用,代码层面因为都是标准 SMTP + env 驱动,**切换发件邮箱不用改代码,只改 `.env`**。
