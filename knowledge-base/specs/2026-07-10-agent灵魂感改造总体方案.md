## 1. 一句话诊断

**当前 Agent 是"规则引擎 + LLM 兜底",不是 Agent。** `Manus.think()` 用约 650 行 if/elif(manus.py:1546-2213)替 LLM 做完了意图识别、工具选择、参数构造、流程编排;真正的 ReAct loop(`ToolCallAgent`,能力齐全)只在 `Intent.UNKNOWN` 兜底时生效。工具是端到端焊死的大颗粒(`send_resume_email` 一个 execute 焊死"渲染 PDF + 固定留言 + SMTP"),没有中间产物、不可组合;新增一个工具要动 14+ 文件、30+ 触点,漏一处白名单就静默丢数据。**要有灵魂感,核心不是加功能,而是把控制权从 if/else 交还给 Agent loop + 细粒度可组合工具 + 协议化的人机确认。**

## 2. 目标画像:改造后「发邮件」应该长什么样

用户说:"帮我投腾讯后端岗,发到 hr@tencent.com"。

1. Agent 发现缺信息 → 调 `ask_user` 结构化提问("语气正式还是亲切?")——**主动澄清,不是正则猜**
2. Agent 自主编排:调 `write_cover_letter`(针对腾讯后端写求职信,草稿落**会话工作区**,前端出**可编辑卡片**)→ 调 `render_resume_pdf`(产出 pdf artifact)
3. Agent 调 `send_email(to, subject, body_ref, attachment_refs)` → 该工具标记 `requires_approval`,**运行时挂起**,SSE 推确认卡(收件人/主题/正文全部**可编辑**)
4. 用户顺手把正文改软一点 → 点「确认发送」→ 带**改后参数**恢复执行 → 发出
5. 中途用户说"语气改软一点" → Agent 只重跑 `write_cover_letter`,不整个流程重来

对照业界四不变式(产品调研 §6.3,Anthropic/OpenAI/Gemini 全部一致):① 草稿是落盘的中间产物;② 附件渲染是独立工具;③ 确认由**运行时挂起**保证,模型想绕也绕不过;④ 确认界面允许编辑草稿,改完的版本才是被发送的版本。

## 3. 总体架构原则(五条)

1. **编排交给模型循环,写死的部分只做护栏不做主干**(Anthropic workflow-vs-agent 分野)。if/elif 不是删光,而是降级:快路径只保留零歧义纯加速(问候、加载简历),且只"建议"不"拦截"。
2. **中间产物是一等公民**:统一落会话工作区(named artifacts),既是下一步工具的输入,也是用户编辑的对象(Manus 文件系统即上下文)。
3. **确认下沉到协议层**:`requires_approval` 是运行时语义(挂起→确认卡→带改后参数恢复),不是 prompt 里求模型自觉(OpenAI `needsApproval` / Claude Code `canUseTool`)。
4. **事件自描述,注册零白名单**:`{type, payload}` 信封无条件透传,前端按 type 查注册表渲染,未知 type 走通用兜底卡。新工具的接入成本收敛到 2 处(后端工具类 + 可选前端卡)。
5. **澄清前置、自验后置**:开工前结构化提问(`ask_user`),交付前校验(附件存在性、PDF 非空),失败原样留在上下文让模型自己改参重试。

## 4. 分阶段路线图

> 依赖关系:Phase 0 是地基(没有它,试点新增的 3-4 个小工具每个又是 30+ 触点);Phase 1 直接兑现用户痛点;Phase 2 治本(think 解耦);Phase 3 补体验与底座。每阶段独立可验收、可上线。

**解决**:瓶颈 P0-1(30+ 注册触点)、P0-2(双白名单静默丢数据)、P0-3(前端每工具一套簿记)。

| 改动 | 位置 |
|---|---|
| `ToolResult` 增加一等 `structured` 字段(`{type, payload}`);`system` 字段兼容过渡 | `backend/agent/tool/base.py` |
| `_store_structured_tool_result` 6 段工具名 if 塌缩为 1 段通用解析,无条件存;解析失败记日志不静默 | `backend/agent/agent/toolcall.py:86-243` |
| SSE 层删工具名白名单,`tool_result` 一律带 structured 透传;resume_updated/resume_patch 等领域事件改由 `type` 驱动 | `backend/agent/web/streaming/agent_stream.py:1069-1122` |
| 前端 `CardRegistry: Record<type, Component>` 按 type 分发;未知 type 通用 JSON 卡兜底;统一一份 `structuredEvents` state + 一个 upsert/rebind/清理管线,替代 6 组平行 state | `frontend/src/hooks/agent-chat/useToolEventRouter.ts` + `CocoChat.tsx:641-686, 1046-1131, 2230-2236` |

### Phase 1 · 邮件试点:拆工具 + 工作区 + 运行时确认(用户痛点主战场)

**解决**:B2 工具不可组合、P1-5 prompt 约定式确认、I4/I5 交互僵化;兑现 §2 目标画像。

| 改动 | 位置 |
|---|---|
| `ResumeDataStore` 旁立/泛化为 `SessionWorkspace`:`put(kind, content) -> artifact_id` / `get(id)`,简历只是一种 kind;写入即推 `artifact_updated` 事件 | `backend/agent/tool/resume_data_store.py` |
| 拆 `send_resume_email` 为三件套:`write_cover_letter(target, tone) -> draft artifact`、`render_resume_pdf() -> pdf artifact`(复用现有链路;顺带把 `_prepare_latex_content/_compile_pdf_bytes` 提为公开服务函数,结束越权 import)、`send_email(to, subject, body_ref, attachment_refs)` 只管发 | `backend/agent/tool/send_resume_email_tool.py` 拆分、`backend/routes/pdf.py` |
| 工具元字段 `requires_approval: true`;`toolcall.py` 执行前命中即挂起,SSE 推 `approval_request`(含参数 + 可编辑字段声明);用户批准(可改参数)后恢复执行;删 `confirm` 参数与 description 文字约定 | `backend/agent/tool/base.py`、`backend/agent/agent/toolcall.py`、`backend/agent/web/routes/stream.py`(接收 approval 响应) |
| 前端通用 `SchemaFormCard`(可编辑表单 + 确认/取消),替代只读 SendEmailConfirmCard 与"替用户打字"按钮 | `frontend/src/components/agent-chat/`、删 `CocoChat.tsx:4206-4212` 文本回发 |
| `ask_user` 升级为结构化澄清工具(问题 + 2-4 选项 schema),前端通用问题卡;邮件流程首用 | `backend/agent/tool/ask_human.py`、前端新卡 |
| 邮件意图不再走 think() 直调,落 ReAct loop 让 LLM 自主编排三件套 | `backend/agent/agent/manus.py`(邮件分支摘除) |

**验收**(GLM 试点标准 + 四不变式):"帮我投腾讯,邮箱 x@y.com" → Agent 追问岗位/语气 → 写针对性求职信(卡片可编辑)→ 附简历 PDF → 弹可编辑确认 → 改正文后发送,收到的邮件正文是改后版本;全过程事件轨迹里的每个 tool call 都是 LLM 真实发起;绕过确认直接发送在运行时不可能发生(单测覆盖)。

**风险**:挂起状态目前只能进程内存(P2-11),试点期绑定 session 可接受,落库放 Phase 3;ReAct 编排比直调多 2-3 次 LLM 往返,需把 max_steps 关键词二值逻辑(agent_stream.py:525-528)放宽为按任务预算。

### Phase 2 · think() 渐进解耦:控制权归还(治本)

**解决**:B1/B3(650 行调度器架空 loop)、P0-4(意图三层混杂)、P1-8(伪造工具)。

策略 = GLM「Dify Agent Node」渐进思路 × 我方「按意图灰度摘除」:
- 高变异分支逐个"工具化下沉":`ANALYZE_RESUME` 两阶段流程收进一个真实注册的 `diagnose_resume` 工具(内部仍可并行调 analyzer,但事件轨迹真实,伪工具 get_resume_detail/resume-diagnosis 退役);`OPTIMIZE_SECTION` 的枚举树同理收进 `optimize_resume` 工具,"改哪段"交给 LLM 在 context 里理解
- 低变异快路径(GREETING/LOAD_RESUME)保留,但从"拦截"降级为"建议"(不再有决定权、不改写用户消息)
- `prompt/manus.py:84-159` 的 cv_editor 使用手册迁入工具 schema/description;`%%SUGGESTIONS%%` 魔法字符串改用已存在的 SuggestionsEvent
- 前端三个 fast-path(问候胶囊/粘贴导入/固定话术)改为建议 chips,不拦截

**验收**:同一句话从任何入口进来行为一致;"优化第一段并翻译成英文"这类组合请求不再被拆丢;事件流 = 真实执行轨迹;规则层代码量下降 ≥60%。

### Phase 3 · 体验与底座(并行迭代)

- **计划外化 + 复述**:多步任务先产出 checklist(SSE 推前端、可增删),每步打勾,计划文本每轮追加上下文尾部(治 30 条滑窗遗忘 + 过程黑盒)
- **记忆分层**:用户级 Knowledge(求职方向/语气偏好/称呼,低频更新每会话注入)+ playbook 按需加载;评估 EntityMemory 617 行死代码是接入还是删除
- **结构化流式协议**:原生 reasoning/content 分离替代 Thought/Response 正则(方向 E),删表演性 sleep 450ms 与 f-string 假思考
- **交付自验**:发送前校验附件存在/PDF 非空;失败 traceback 原样回灌 observation
- **工程清偿**:CocoChat 巨石拆分(P1-6)、挂起/限频/会话态外置存储(P2-11)、诊断评分公式(竞争力=质量+8)换真实评估

## 5. 明确不做的事

1. **不上多代理框架**:简历场景是"深而窄"任务,单 Agent + subagent 按需隔离足够(Cognition 主张 + Anthropic "start simple");
2. **不一次性删 think()**:按意图灰度摘除,每摘一个跑回归;
3. **不把问候/加载简历也交给 LLM 全跑**:纯加速快路径保留(降级为建议),延迟敏感场景不为"纯度"牺牲体验;
4. **不自研可视化 DAG 编排**(Coze/Dify 那套):我们是产品不是平台,LLM-in-the-loop 已覆盖需求。
