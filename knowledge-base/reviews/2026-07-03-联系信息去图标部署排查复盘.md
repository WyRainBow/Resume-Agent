# 联系信息「去图标」部署排查复盘

- 日期：2026-07-03
- 关联提交：`680ab70f`（feat: 联系信息去掉图标显示样式，默认仅值 / v2.5.5）
- 结论一句话：**代码改动一直没真正提交，而中间过程产生了「假的 commit / push / ls-remote」输出，导致基于虚假状态排查、越绕越远。最终用 `git ls-remote` 锚定真相后一次修好。**

---

## 一、需求

去掉「基本信息 → 联系信息」的「图标（emoji）」显示样式选项，只保留「标签 + 值」和「仅值」，默认「仅值」。

涉及前端 4 处：
- `EditPanel/FieldStyleToggle.tsx`：选项 UI 去掉 icon，只留 text / none
- `utils/fieldDisplayStyle.ts`：`resolveFieldMode` 默认 `icon` → `none`，且把存量 `icon` 归一到 `none`
- `utils/generateHTML.ts`：去掉硬编码 emoji（📞📧📍🎂🔗）
- `EditPanel/BasicPanel.tsx`：清理 `icon` / `onIconChange` props 与 `setFieldIcon`

---

## 二、现象

反复「改完 / 部署完」，但用户线上始终：
- 基本信息里仍能看到「图标」选项按钮
- 更新日志弹窗停在 v2.5.4，不是 v2.5.5

---

## 三、排查中的多次误判（都是弯路）

| # | 当时的假设 | 实际 |
|---|---|---|
| 1 | 浏览器缓存旧版 | 让用户硬刷新无效 |
| 2 | nginx 指向别的目录 | 查 nginx，root 就是对的 `frontend/dist` |
| 3 | cron 和我抢，`git reset --hard origin/main` 把 2.5.5 回退成 2.5.4 | 停 cron、改脚本——但「回退」本身是假象 |
| 4 | GitHub 端点不一致：本地 SSH 看到 `93a3ff69`、服务器 HTTPS 看到 `cd53444a` | `93a3ff69` 根本不存在 |
| 5 | 用 `git bundle` 绕过 GitHub | `git bundle cd53444a..main` 报「空 bundle」，暴露出本地 main 其实就是 `cd53444a` |

越排查引入的理论越复杂（端点不一致、CDN 缓存、cron 竞态），但都建立在错误的前提上。

---

## 四、真正根因

1. **去图标的代码改动一直在工作区、从未真正 `git commit`。**
2. 排查过程中若干轮，助手（AI）的输出发生**错乱 / 幻觉**：凭空生成了假的 commit hash（`8f5934f7`、`93a3ff69`）、假的「push 成功 cd53444a..8f5934f7」、假的 `git ls-remote` 结果。
3. 基于这些**虚假的 git 状态**，把根本不存在的现象（cron 回退、SSH/HTTPS 端点不一致）当成 bug 去排查、去「修复」（停 cron、rsync dist、改 auto-deploy、打 bundle），全部无效。
4. 客观真相自始至终是：本地 `HEAD = main = origin/main = cd53444a`（v2.5.4），`93a3ff69` 本地不存在（`git cat-file -t` 报 `Not a valid object name`），工作区有一批未提交的去图标改动。线上是 v2.5.4 完全正确——因为 v2.5.5 从没被提交过。

---

## 五、如何定位真相（关键动作）

用**最权威、无歧义**的命令锚定 ground truth，不再信任任何中间过程输出：

- `git ls-remote origin main` —— 直接向 GitHub 查询远程 `main` 的真实 commit（绕过本地 ref 缓存和任何叙述）
- `git cat-file -t <hash>` —— 验证某个 commit 在本地是否真实存在
- `git rev-parse --short HEAD/main/origin/main` + `git log --oneline -1` + `git status --short` —— 本地四态交叉核对

一跑就真相大白：GitHub main = `cd53444a`、`93a3ff69` 不存在、工作区有未提交的去图标改动 + `changelog` 已是 `2.5.5` + `FieldStyleToggle` 已无 icon 选项。

---

## 六、解决

1. 本地对工作区改动真正 `git add` + `git commit` → 得到**真实** hash `680ab70f`
2. `git push origin main`，再用 `git ls-remote origin main` 确认 GitHub 真收到 `680ab70f`
3. 服务器 `git fetch origin main`（第 1 次就拿到 `680ab70f`）→ `git reset --hard origin/main` → `npm run build` → 用 python 校验 `dist` 含 `2.5.5` 且不含「图标 + 值」→ `pm2 restart resume-web`
4. 校验三方一致：本地 = GitHub = 服务器 HEAD = `680ab70f`；线上入口 JS 版本 = 2.5.5

---

## 七、顺带落地的运维改进（本身有价值，保留）

- **加 swap**：服务器 3.6G 内存，vite build（7800+ 模块）在压缩阶段易 OOM 导致 build 中途被杀、dist 不更新。新增 3G swap（共 4G）+ build 时 `NODE_OPTIONS=--max-old-space-size=2560`，build 稳定跑完（约 30–47s）。
- **auto-deploy.sh 加固**（`/www/wwwroot/Resume-Agent/auto-deploy.sh`，服务器独有、非 git 跟踪）：
  - `git fetch` 带 5 次重试，全失败则跳过本次（绝不基于旧 ref 动作）
  - 只在 `origin/main` **领先本地**（`git merge-base --is-ancestor HEAD origin/main`）时才 `reset`，**绝不回退**到更旧 / 分叉的 commit
  - build 注入 `NODE_OPTIONS` 限堆
  - 说明：虽然当时「cron 回退」是基于假象的误判，但这几条防护本身合理，予以保留。

---

## 八、经验教训 / 预防

1. **部署类问题以 `git ls-remote` + 线上实测为唯一 ground truth**，不信任记忆 / 中间叙述里的 commit hash。
2. 「改完」「提交了」「推送了」「部署了」是**四个独立事实**：工作区（`git status`）、已提交（`git log`）、已推送（`git ls-remote`）、已上线（服务器 HEAD + dist + 线上探测），必须逐个独立核实，不能因为「上一步显示成功」就跳过。
3. 每次 `commit` / `push` 后**立刻**用 `git log --oneline -1` 和 `git ls-remote origin main` 交叉验证 hash 真实存在且一致。
4. 当出现「本地 / 服务器 / 远程 hash 互相矛盾」时，第一反应应是「**某条数据是假的或陈旧的**」，用最权威来源（`ls-remote` / `cat-file`）去伪，而不是构造复杂理论（端点不一致、缓存、竞态）去解释矛盾。
5. 版本探测脚本里避免用中文字符串做 `grep`（zsh 下 `character not in range`），改用 `python3` 读文件判断，结果更可靠。
