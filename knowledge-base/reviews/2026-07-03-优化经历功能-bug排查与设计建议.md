## 1. 触发场景（用户反馈的三个问题）

1. **「优化一下实习」会把实习和项目经历都识别进来** —— 用户认为大多数人说的是「帮我优化一下实习经历」这种精确命令，实习和项目应当拆开。
2. **没有「优化技能」的能力** —— 说「优化一下专业技能」，AI 回「抱歉，未能准确匹配「专业技能」。请从下方选择要优化的经历：」，只能从经历里挑。
3. **多段实习时的交互 / 后续引导（产品设计问题）** —— 用户有多段实习，说「优化实习经历」时会弹按钮让选哪一段（用户不确定这个选择是否是好设计）；并希望「优化完一段后 AI 主动问是否优化下一段」；或者干脆提供一个「优化我的简历」的通用功能，各部分一起优化。

## 2. 「优化」链路现状（代码级）

### 2.1 意图路由（`backend/agent/application/conversation/conversation_state.py:_detect_agent_intent` 616-648）

| 用户措辞 | 判定意图 | 代码位置 | 结果 |
|---|---|---|---|
| 「全面/整体/全局优化」 | `FULL_OPTIMIZE` | `:627-628` | 批量，但实际只覆盖工作经历 |
| 量化/润色/突出/改进/丰富/改写/重写 + section | `EDIT_CV` | `:631-635`（`SEMANTIC_EDIT_KEYWORDS`） | 走 `cv_editor_agent` 字段编辑 |
| 含「优化」+ 不是新增经历 | **`OPTIMIZE_SECTION`** | `:638-639` | 截图里这条，主路径 |
| 分析/评估/诊断 | `ANALYZE_RESUME` | `:641-646` | 体检诊断 |

上游 `_extract_section`（`:650-662`）其实能把「专业技能」抽成 section=「技能」、把「教育」抽成「教育背景」，**但这个 section 信号在 OPTIMIZE_SECTION 分支被丢弃、没用于路由**。

### 2.2 可优化的 section 范围（只有三类）

`backend/agent/utils/experience_entry.py:_iter_optimize_sections`（206-219）只枚举：

- `experience`（实习/工作，value_field=`details`）
- `projects`（项目，value_field=`description`）
- `openSource`（开源，value_field=`description`）

**技能 / 教育 / 自我评价 / 荣誉奖项都不在可优化范围内**，只能靠通用 `cv_editor_agent` 做字段级替换（`backend/agent/tool/cv_editor.py:15-25` 别名含 skillContent/selfEvaluation/education/awards），**不是** experience 那套按四要素 / STAR 深度改写的 diff。

### 2.3 OPTIMIZE_SECTION 处理（`backend/agent/agent/manus.py:1670-1768`）

1. `resolve_optimize_target(user_input, recent_assistant, snapshot)` 尝试解析出**一个**具体目标（`experience_entry.py:349-402`，公司/岗位模糊匹配）。
2. `list_optimize_targets(snapshot)` = 上述三类的全部条目。
3. 分支：
   - 无目标 + 只有 1 段 → 直接优化那段（`:1731-1732`）。
   - 无目标 + 0 段 → 「当前简历里还没有可优化的实习/工作、项目或开源经历」（`:1686-1694`）。
   - 无目标 + 多段 + 泛化查询 → 澄清列表，引导语「好的！在优化之前，请问您想优化哪一段经历？」（`:1696-1703`）。
   - 无目标 + 多段 + 非泛化 → **「未能准确匹配「X」。请从下方选择要优化的经历：」+ 澄清列表**（`:1704-1729`，「未能准确匹配」在 `:1717`）。
4. 命中单一目标 → `_llm_optimize_section_patch`（`:654-766`）主 LLM 直接产出优化 diff → 入队成 `resume_patch` 卡片。回复 `_optimization_assistant_reply`（`:780-811`）：「已为「X」生成优化对比，请确认」，然后 `state=FINISHED`。

### 2.4 前端建议按钮（`%%SUGGESTIONS%%`）

- 原始标记由**后端** `agent_stream.py` 解析成 `suggestions` SSE 事件下发；前端在 `CocoChat.tsx:1487-1492` 消费 `items`，`StreamingLane.tsx:170-267` 渲染成「下一步建议」按钮。

- 优化 diff 卡 = `ResumeDiffCard.tsx`；「应用」→ `ResumeContext.tsx:63-77 applyPatch` 在**前端本地**按 path 合并 `after` 写回并触发 PDF 重渲染。

## 3. 三个 bug 的根因

### Bug 1 · 「优化实习」把项目也拉进来
- `is_generic_optimize_section_query("优化实习")` 把核心词「实习」判成**泛化**（`experience_entry.py:151-162` 白名单含「实习/工作/项目/开源/…」）→ 不收窄到具体段。
- 收窄不到 → 弹澄清列表 `build_optimize_clarification_suggestions`（`:246-271`），它**无条件枚举 experience+projects+openSource 全部**，不按用户说的「实习」过滤。
- **本质：系统没有「section 类型意图」概念**——「实习」「项目」「开源」被一视同仁地当成「随便优化哪段」，所以混在一个列表里。

### Bug 2 · 「优化专业技能」直接失败
- 技能不是可优化 target（`_iter_optimize_sections` 无 skills）。上游已识别 section=「技能」但 OPTIMIZE_SECTION 分支**丢弃**该信号，只拿去 experience/project/opensource 里找「专业技能」这家「公司」→ 找不到 → 落到「未能准确匹配「专业技能」」（`manus.py:1717`）。
- `skills_analyzer` 只产 issue、不产 `optimized` 文本，会被 `manus.py:774` 过滤 → 即便走 FULL_OPTIMIZE 也生不成技能优化 diff。技能/教育/自我评价/荣誉**全都没有「优化」入口**。

### Bug 3 · 多段选择 UX / 优化下一段 / 整份优化
- 点击「下一步建议」**只填输入框、不自动发送**（`CocoChat.tsx:4515`）——摩擦。
- **没有「优化完一段主动问下一段」**：`_optimization_assistant_reply`（`manus.py:780-811`）只回「已生成，请确认」，随后 FINISHED；一次只优化一段，无队列。
- **对话里没有「一键优化整份简历」**：「优化我的简历」这种整篇请求落到「未能准确匹配「简历」」的经历选择列表；`FULL_OPTIMIZE` 关键词受限（必须显式「全面/整体/全局优化」）且实际只覆盖工作经历。真正的整份一键优化只在另一个界面（Workspace 的 `JdMatchDialog.tsx:115`），对话里没有。

## 4. 刘小排经验映射

| 他的原则 | 落到这里 |
|---|---|
| 问题很具体、核心动作明确、用户一句话就能理解 | 「优化实习经历」就是精确命令，产品该**照字面尊重 section 意图**，而不是反问把实习和项目混着丢回去。用户「拆开」的直觉正确。 |
| 每一屏回答一个问题 + 选择悖论 | 用户只说「实习」却弹实习+项目+活动一大串 → 增加决策成本。选择列表必须**收窄到用户点名的 section**。 |
| 先把一个小问题解决到足够好 | 优化是核心动作，先把「优化某 section」「优化整份」两条主干做扎实；技能这种常见需求**静默失败比不提供更糟**（反对半成品）。 |
| 产品结果直接可见、可反复使用（+ 把低频做成高频） | 「一键优化整份 → 一叠对比卡逐条确认」结果立即可见；「优化完主动问下一段」让用户一段接一段用下去 = 提升单次使用深度。 |

## 5. 建议（按价值排序）

1. **最高价值：把「一键优化整份简历」做成对话里的一等公民（核心动作）。** 说「优化我的简历」→ 对所有 experience/project（+技能）逐段生成优化 diff → 前端呈现成一叠对比卡，支持「全部应用」+ 逐条确认。最贴合「核心动作明确 + 结果直接可见」，也把优化从一次一段的低频操作变高频。（当前「优化简历」反而报错，是明显 bug。）
2. **修 section 类型意图（Bug 1）：** 让「优化实习/项目/开源/技能」各自**收窄到对应 section**；该 section 只有 1 段→直接优化，多段→选择列表**只列这个 section**。上游已识别 section，只是没被用起来。
3. **补技能优化（Bug 2）：** 至少支持「优化技能/专业技能」；对暂不支持的 section 别丢进经历列表，而是明确回应或路由到 cv_editor。技能改写逻辑与经历不同（关键词/清单 vs STAR 条目），需单独设计。
4. **引导式「下一段」：** 每次优化完给几个**自动发送**的建议按钮：「继续优化下一段」「优化其它部分」「优化整份」「就到这」。低摩擦、提高完成率。
5. **选择列表点击即发送 + 加「全部一起优化」项：** 去掉「填进输入框还要手动发」的摩擦；列表顶部加「这些全都优化」。

**对「多段选择是否好设计」的判断：** 选择列表本身没问题，但只在「同 section 内真有多段」时才该出现，且要收窄 + 点击即发 + 带「全部」选项。真正该补的是①的整份一键优化——它才是大多数人想要的默认动作，选择列表是补充、不是主入口。

## 6. 待定的产品方向（未决，等用户拍板）

主方向 4 选项：
- **A（推荐）整份优先 + 分段精确：** 整份一键优化做核心动作，同时修好分段收窄。整份是主入口，分段是精确补充。
- **B 只做分段精确：** 先不做整份，只修 bug（实习只列实习、技能能用、多段收窄+点击即发+引导下一段）。
- **C 只做整份一键：** 只做「优化我的简历 → 各部分一起优化 → 一叠对比卡逐条确认」，分段收窄暂不动。
- **D 只补技能优化：** 范围最小，只解决 Bug 2。

用户想先厘清的点（讨论中）：
1. 整份一键优化怎么呈现才不烦人（review 负担 / 全部应用+折叠逐条）。
2. 分段 vs 整份谁当默认。
3. 技能优化做到什么程度（不报错即可 vs 按岗位真重写/排序）。
4. 「优化下一段」主动询问是否啰嗦。
5. 是否对某个 bug 根因/判断有疑问需再深挖。

## 7. 实施记录（2026-07-03，feature/agent-chat-ux）

**已按方案 A 全量落地**（commit `1f210177` + 透明化迭代）：

- **整份优化按明确程度分两档**（用户反馈「直接跳出全部优化不透明」后加的透明化设计）：
  - soft（「优化简历 / 优化我的简历」范围模糊）→ 先回**优化计划**：「可优化的部分有：实习/工作经历 2 段、项目经历 1 段、专业技能、自我评价（共 5 处）。想全部一起优化、还是先从某一部分开始？」+ 点击即发送的选择按钮。
  - explicit（「优化整份简历 / 全部一起优化」，含点计划里的按钮）→ 直接执行，不二次确认（黄金规则：明确指令不反复确认）。
  - 判定：`experience_entry.detect_whole_optimize_mode`；计划清单：`build_optimize_plan_overview`。
- **整份执行透明化**：`_optimize_whole_resume` 用 `asyncio.as_completed`，每段完成即入队 patch（前端对比卡渐进出现）；完成回复**列出优化了哪几段**（「已优化：自我评价、专业技能、美的集团、腾讯、校园二手交易小程序，共 5 处对比」）而非只报个数。
- **section 收窄（Bug1）**：「优化实习」只列实习段；单段直接优化。`detect_optimize_section_kind` + `list_optimize_targets(kind)`。
- **技能/自我评价优化（Bug2）**：`_llm_optimize_field_patch` 单字段改写（skillContent / selfEvaluation）。
- **优化完引导**：单段优化完追加自动发送建议「✨ 优化整份简历 / 换一段优化」。
- **前端**：建议按钮点击即发送（不再只填输入框）；澄清列表顶部「✨ 全部一起优化」。
- **FULL_OPTIMIZE**（全面/整体优化）改走整份优化（原 resume_optimizer 只覆盖工作经历）。

**SSE 实测**（带 2 实习 + 1 项目 + 技能 + 自评的简历）：
| 输入 | 结果 |
|---|---|
| 优化简历 | 计划清单 + 5 个选择按钮，0 patch ✅ |
| 优化我的整份简历 | 直接 5 处对比，回复列出各段 ✅ |
| 优化实习 | 只列 2 段实习 + 全部选项 ✅ |
| 优化技能 | 1 patch + 引导建议 ✅ |
| 优化项目（单段） | 直接优化 1 patch ✅ |
