## 一、方法论：刘小排里跟「交互反馈」直接相关的三把尺子

| 原则 | 一句话 | 落到交互上 |
|---|---|---|
| **买时间** | 每一步只争取下一步；别让用户在无反馈里干等 | 任何 >1s 的等待都要有「活的」反馈，且尽量给出**边界感**（还要多久/还剩几步） |
| **结果直接可见** | 做完立刻看到效果 | 结果一产出就呈现（右侧预览、对比卡、before-after），不让用户再点一下才看到 |
| **静默失败比不提供更糟** | 每个失败都要让用户知道 + 能重试 | 不 silent catch；失败要有明确文案 + 一键重试，不让用户重新组织语言 |

**判据**：产品里每一个「用户发起动作 → 系统在忙 → 结果出来」的环节，问三句——
① 忙的时候用户看到的是「活的反馈」还是「死等/灰按钮」？
② 结果出来是「自动可见」还是「要用户再操作」？
③ 万一失败，用户「知道且能一键重试」还是「一脸懵」？

## 二、已经做对的（作为参照样板）

- **导入解析 → 思考动画**（v2.12.1，本次）：图片/文本/文件三入口的解析等待都用 `ThinkingIndicator`，并隐藏了占位消息上不该有的复制/赞/踩。✅ 买时间
- **整份优化 → 对比卡渐进出现**（v2.7.0）：`as_completed` 让每段优化完就出一张卡。✅ 过程渐进可见
- **保存状态常驻指示器**（v2.9.0）：保存中…/已保存/保存失败。✅ 结果可见 + 失败不静默
- **PDF 预览不闪烁 + 保留旧版**（v2.12.0）：重渲染时旧 PDF 持续可见到新的就绪。✅ 结果直接可见
- **SSE 出错「重新发送」、PDF 报错可重试、润色失败重试**（v2.8.0）：✅ 静默失败比不提供更糟
- **流式思考占位**（`StreamingOutputPanel` 用 `ThinkingIndicator`）：思考首字未到时星芒占位。✅ 买时间

**结论**：基础反馈能力已经不错。下面是仍有「死等/无边界/结果需二次操作/静默失败」的具体缺口。

## 三、可优化点（按价值排序，均为交互反馈类，不含已暂缓的支付/移动端）

### P0-A · 整份优化缺「进度边界感」——最该做，且最贴本次思路

**现状**：用户说「优化整份简历」后，后端 `_optimize_whole_resume` 直接跑 ~30s，中途靠对比卡逐张冒出来，但**没有「共 N 段、已完成 M」的进度**，也没有开场的「正在逐段优化（共 8 段）…」预期。用户面对的是「一张张卡慢慢冒，但不知道还有几张、还要等多久」。

**踩了哪条尺子**：买时间——长等待**没有边界感**是焦虑主因；刘小排强调每一步都要让用户知道「进行到哪了、还要多久」。

**建议**：
- 开场立即出一条**带进度的思考消息**：「正在逐段优化你的简历（共 N 段），完成一段出一张对比卡…」——N 用 `list_optimize_targets` + 技能/自我评价数，**开跑前就能算出**。
- 每入队一张卡就更新进度：「已完成 3/8…」。可复用 `ThinkingIndicator` + 一个 `x/N` 计数，或做个细进度条。
- 技术落点：后端在 `_optimize_whole_resume` 开头先发一个 `optimize_plan`/进度 SSE 事件（总数 N），每段 `as_completed` 后发 `optimize_progress`（done/total）；前端把这条思考消息随事件更新。
- **价值**：把「30s 黑箱等待」变成「8 段进度条」，是本次「导入思考动画」思路的自然延伸，且优化是核心动作，收益最大。

### P0-B · 发消息 → 首个 SSE 之间的「死等窗口」

**现状**（见 [[2026-07-06-Agent对话交互优化分析]] P1-4）：用户发消息后，经历「网络 RTT + 后端建会话 + 首次 LLM 调用（2-5s）」才有第一个事件。这段时间只有输入框变灰（`isProcessing`），**没有思考动画**。首次响应延迟感强。

**踩了哪条尺子**：买时间——最该有反馈的恰恰是「刚点完、还没任何回音」的头几秒。

**建议**：`sendMessage` 一旦 `isProcessing=true`，立即在对话流末尾插入 `ThinkingIndicator`（「思考中…」），首个 `thought/answer_chunk` 到达再替换。项目已有组件，改动小、体感立竿见影。

### P1-C · 全站失败态统一「可重试」审计

**现状**：部分失败已可重试（SSE 重新发送、PDF 报错、润色失败），但仍有零散的 `catch { toast.error }` 后**要用户重新发起**的点（如图片识别失败、导入解析失败、评分失败）。

**踩了哪条尺子**：静默失败比不提供更糟——失败不仅要提示，还要**能一键重试**，不让用户重新组织输入。

**建议**：盘一遍所有 `upload-image/upload-pdf/parse/score` 失败分支，失败消息统一带「重试」按钮（重发同一份文件/同一段文本），而不是只弹 toast。断线场景加输入框上方 banner（分析文档 P3-13）。

### P1-D · 长回答跳过打字机，别让「结果可见」被动画拖慢

**现状**（分析文档 P3-15）：Agent 长回答（几百字）走打字机效果，用户要等很久才看到完整内容。

**踩了哪条尺子**：结果直接可见——动画本意是「活」，但过长的打字机反而**延迟了结果可见**，与原则相悖。

**建议**：长回答（>200 字）跳过打字机直接整段显示；短回答保留打字机的「活」感。

### P2-E · PDF 渲染进度阶段化（锦上添花）

**现状**：闪烁已修（保留旧 PDF）。但渲染进度文案仍是笼统的「正在渲染 PDF...」。

**建议**（可选）：把进度分阶段（准备数据 → 排版编译 → 生成预览），让 3-8s 的 LaTeX 编译更有过程感。优先级低，因为旧 PDF 已保留、体感的「空白焦虑」已解决。

## 四、下一批推荐顺序

**P0-A（整份优化进度）> P0-B（首响思考动画）> P1-C（失败可重试审计）> P1-D（长回答跳打字机）**。

理由：A 是核心动作（优化）上的最大等待、收益最大，且是本次「loading 互动」思路的正统延伸；B 改动极小、体感立竿见影；C 是「静默失败」这条硬原则的收口；D 是顺手的阅读体验修正。E 作为可选锦上添花。

**均不涉及支付闭环 / 移动端**（两者已明确暂缓，见路线图「暂缓」节）。所有落点复用已有 `ThinkingIndicator` / SSE 事件 / toast+确认框，不新建平行实现。
