# Agent 架构下一步优化方向(初步梳理)

- **日期**:2026-07-11
- **分支**:`feature/agent-arch-wave2a`
- **背景**:`knowledge-base/reviews/2026-07-11-agent-architecture-audit-and-paradigm-research.md` 的架构审计"优先级 3 件事"已全部落地(commit `46b57dcf`/`d3b373e2`),manus.py 1578→794 行,`resume_use_cases.py`/规则引擎/`experience_entry.py` 意图猜测函数悉数清空,`_llm_first_routing_enabled` 开关连同回退能力物理删除。真实 SSE 对话测试(诊断/编辑两条链路)验证了清理没有引入回归,且 LLM-first 带来了实质性行为改善(诊断一步到位、编辑场景 LLM 主动拒绝编造数据反问用户)。本文档是"接下来还能做什么"的初步判断,供后续 opus agent 深入展开任务清单和具体方案参考,不代表已批准的实施计划。

## 一、遗留的文件级决策(低工作量,随时可做)

`IntentRouter`(119 行)/`ToolInvocationBuilder`(54 行)在 Task B 清理后都变得很薄,`ToolInvocationBuilder` 基本只剩 `build_apply_optimization` 一个方法。Task B 的 opus review 已经指出这个情况,但按计划要求"文件级合并/删除属于范围外决策,只报告不处理",一直悬着没定。需要评估:保持独立(职责边界清晰,后续可能还会长出新内容)还是合并进更少的文件。

## 二、"灵魂"相关的建议一项都还没落地

两轮外部调研(2024-2025 + 2026 补充)给出的具体手法——价值观人设(替代规则清单)、SOUL.md 式渐进披露、generator/evaluator 分离——目前一个都没有系统性实施。现在 system prompt 里的人设(那个叫"coco"的角色)具体是怎么写的还没有系统评估过,值得对着调研报告里"规则产生服从,价值观产生性格"这条标准检查一遍,看有没有"过度中立/怕出错就不给意见"这类问题。今天实测编辑对话时 LLM 主动拒绝编造数据、反问用户,这本身就是"有主见"的好表现,说明现在的人设设计可能已经不差——建议先系统评估现状,再决定要不要动,不要预设"一定要大改"。

## 三、2026 新增的"验收契约"机制,现在完全没有

Anthropic 2026-03/04 两篇官方文的核心观点是"确定性预算该花在验收契约/状态管理/权限隔离/错误恢复上,不该花在猜意图上"。删完规则引擎后,think() 里对"什么算优化完成/编辑成功"这件事完全没有显式校验层——比如 `cv_editor_agent` 改完之后,有没有检查生成的内容是不是真的合理(不是格式/HTML 规范化这类既有的确定性校验,是"改得好不好"这层语义质量)。建议先不预判性地加,等确实观察到质量问题(比如 LLM 瞎改、幻觉数据)再考虑要不要引入一个轻量 evaluator(且必须和 generator 分离,参考调研报告里 Reflexion 的"自评自己的错误"这个坑)。

## 四、垂直合规:一致性校验 + 反"AI 味"检测

2026 补充调研里的新发现:招聘方在明确排查"AI 生成简历的逻辑矛盾"(日期重叠、不可能的晋升路径等)。目前简历优化 Agent 完全没有这类校验能力。这是纯新功能,不是架构清理,如果要做需要走 `/brainstorming` 单独立项。

## 五、初步优先级判断

排序:一(顺手小事)> 二(先评估现状,不一定要动)> 三/四(等有明确信号再做,不预判性加)。当前判断是"方向已经对、代码已经干净",接下来更多是产品打磨而非架构手术,不需要像 Wave 2a 那样大动干戈。
