# Agent 架构现状与瓶颈分析

> 2026-07-10 · 只读架构审查。范围:`backend/agent/` 全链 + `frontend/src/pages/AgentChat/` + `frontend/src/components/agent-chat/` + `frontend/src/hooks/agent-chat/`。
> 问题意识:Agent「不好用、功能僵硬」——很多行为疑似硬编码;工具无法动态组合(例:发邮件只能发固定文案 + PDF,做不到「先写求职信 → 用户确认 → 作为正文发送」)。本报告回答三个问题:**硬在哪、为什么硬、要变软动哪里**。

---

## 1. 架构全景

系统名义上是 ReAct Agent(Manus 继承 ToolCallAgent 继承 ReActAgent 继承 BaseAgent,`backend/agent/agent/base.py:118-156` 的 step 循环),但实际运行时,**LLM 的自主决策被三层"抢跑"包裹**:

1. **前端抢跑**(CocoChat.tsx):问候、"创建简历"意图、粘贴简历长文本,三类输入在前端就被拦下,直接插入写死的 assistant 消息或走 `/api/resume/parse`,**根本不发给 Agent**。
2. **规则层抢跑**(ConversationStateManager + domain/intent):问候/加载简历/简单编辑/新增经历四类 fast-path 命中后 `should_skip_llm=True`,直接指定工具与参数。
3. **Manus.think() 抢跑**(manus.py:1546-2213,667 行的 think):ANALYZE_RESUME / OPTIMIZE_SECTION / FULL_OPTIMIZE 三个意图整个业务流程写死在 think() 里——自己伪造 ToolCall、自己拼报告、自己算分数,LLM 只负责最后的文本润色。**只有 UNKNOWN 意图才真正走到 `super().think()` 让 LLM 自选工具**(manus.py:2213)。

**一句话概括**:这不是一个"LLM 驱动、工具可组合"的 Agent,而是一个**用 if/正则/白名单把十几条固定业务流程焊死、LLM 只在缝隙里填文案**的规则引擎;每条链路(工具 → structured_data → SSE → 前端 state → 卡片)都是一根独立拉通的专线,专线之间不共享任何机制。

## 3. 工具机制现状

**注册**:工具是 Python 类(`tool/base.py:80` BaseTool,name/description/parameters + execute),在 `Manus._build_tool_collection`(manus.py:141-171)里以硬编码数组实例化,经 `ToolCollection`(tool/tool_collection.py:17-19)建 name→instance 映射;`_inject_tool_context`(manus.py:180-188)靠 `hasattr` 鸭子注入 session_id/shared_state/user_id。

**被 LLM 选择**:`to_params()` 转 OpenAI function-calling schema 后传给 `llm.ask_tool`(toolcall.py:321-343)。但如 §1 所述,大多数高频意图在到达 LLM 前已被规则层/think() 抢答,LLM 真正自由选工具的窗口只有 UNKNOWN 意图。

**结果回传**:`ToolResult{output, error, system, base64_image}`(tool/base.py:40-47)。`output` 进对话 memory(截 `max_observe=10000`,manus.py:97);**结构化数据没有一等通道**——约定塞在 `system` 字段的 JSON 字符串里,由 toolcall.py:86-243 的白名单逐工具解析暂存到 `_tool_structured_results`,再由 agent_stream.py:1070 的第二份白名单捞出来挂到 SSE `tool_result.structured_data`,最后由前端 useToolEventRouter 的第三层按名分发。**三层每层都是按工具名 if,任何一层漏了都静默丢数据。**

**为什么工具之间无法组合——三个结构性原因**:

1. **端到端闭环大颗粒**。每个工具都把"取数据 → 业务处理 → 副作用 → 面向用户的文案"焊死在一个 execute 里。以 `send_resume_email_tool.py:execute` 为例,一个函数里串了:收件人校验(:83-85)→ 从 ResumeDataStore 拿简历(:87-89)→ 生成默认主题/留言(:92-93)→ 确认文案(:96-113)→ 查库校验 admin(:141-143)→ 取 SMTP 凭证(:145-156)→ 限频(:159-160)→ **内部直接 import pdf.py 私有函数渲染 PDF**(:189-199)→ SMTP 发送(:168-184)。"渲染 PDF"“写文案"“发送"三件事没有任何一件是独立可调用的能力。cv_editor_agent_tool.py:149-321 同理:路径规范化、富文本转换、diff 构造、数据库持久化、patch 事件全在一个 execute 里。
2. **没有可传递的中间产物**。工具返回的是"给人看的字符串 + 给前端卡片的 JSON",不是"给下一个工具用的对象"。会话里唯一的共享区 `ResumeDataStore`(tool/resume_data_store.py)只存两样东西:简历 dict 和 session JD(:172-180)。一段生成的求职信、一次搜索结果、一份分析结论,都没有地方放、没有 ID 可引用,下一个工具调用时只能指望 LLM 从对话文本里"抄"出来——而 memory 还被 max_observe 截断和 30 条滑窗清洗过。
3. **多步执行预算与终止机制敌视组合**。max_steps 默认 5(agent_stream.py:528),LLM 一旦输出纯文本就 auto-terminate(toolcall.py:406-410),Manus 各 fast-path 动辄直接 `state=FINISHED`。一个"搜索 → 起草 → 确认 → 发送"的四步计划,在这套机制下第一步后就很容易被判定"已完成"。

## 4. 「发邮件只能发 PDF」根因分析

目标场景:**"帮我给 hr@x.com 投简历" → Agent 起草求职信 → 用户在卡片里改两句、点确认 → 求职信作正文 + 简历 PDF 附件发出**。逐环节对照现状:

| 需要的能力 | 现状 | 证据 |
|---|---|---|
| ① 起草求职信(独立能力) | 不存在。没有任何"生成文案"类工具;写作能力只以 prompt 附录形式存在于优化链路(OPTIMIZE_SECTION_LLM_ADDENDUM) | `prompt/manus.py:249-272`;`tool/` 目录无相应工具 |
| ② 正文可编辑的确认 UI | SendEmailConfirmCard 是**纯只读**卡片:Row 只展示 to/subject/message,按钮只有确认/取消,没有任何输入框 | `frontend/src/components/agent-chat/SendEmailConfirmCard.tsx:20-27`(Row 组件)、全文 104 行无 input/textarea |
| ③ 确认动作回传结构化数据 | 点"确认"= 替用户发一条文本"确认发送",走完整的意图识别+LLM 轮次;改过的正文(如果能改)无通道传回 | `CocoChat.tsx:4206-4212` |
| ④ 第二轮工具调用还原第一轮参数 | 完全靠 LLM 从上一轮对话文本里回忆 to_email/subject/message 并再次调用工具;服务端不存 pending 状态,description 软约束要求这次带 confirm=true | `send_resume_email_tool.py:45-50`(约定)、工具内无任何 pending-state 存取 |
| ⑤ 把"用户确认过的正文"作为 body 发送 | message 参数每轮由 LLM 重新生成;即便用户口头改了措辞,最终发送的是模型第二轮自己拼的 message,默认兜底又回到写死的"您好,附件是我的简历…" | `send_resume_email_tool.py:92-93`(兜底)、`:124-131`(_do_send 只认本轮入参) |
| ⑥ PDF 与发送解耦(比如只发正文不带附件,或附件换一份简历) | PDF 渲染硬编码在 _do_send 内部(`_render_pdf` 直接 import `backend/routes/pdf.py` 的 `_compile_pdf_bytes/_prepare_latex_content/_resolve_template_dir` 三个私有函数),附件必发、必是当前会话简历 | `send_resume_email_tool.py:162-166, 189-199` |

**根因归纳**(按因果顺序):

1. **工具颗粒错误**:send_resume_email 是"业务终点"而不是"能力"。文案(compose)、渲染(render_pdf)、发送(send)三个能力焊死后,LLM 无法重排它们的顺序或替换其中一环——这是"只能发固定 PDF"的直接原因。
2. **没有会话工作区**:即使拆出"起草求职信"工具,产出的信也无处存放、无法被 send 工具按引用取用(ResumeDataStore 只认简历,resume_data_store.py:20-25)。
3. **没有结构化的人机确认协议**:ask_human 工具(tool/ask_human.py:23-35)只是把问题文本原样返回,不阻塞、不带 schema、前端无对应表单;于是发邮件只好在 description 里自创 confirm 约定(#35),前端只好手写一张只读卡(#45)。每个需要确认的新工具都要再发明一遍。
4. **确认往返会被抢跑层劫持**:用户点"确认发送"发出的文本要再过一遍前端 fast-path、规则层、think() 抢跑(§1 三层);"取消发送"里带"发送"字样、"确认"两个字接近 optimize-confirm 正则(manus.py:2392),路由到别处的风险真实存在。
5. **多步预算不够**:起草 → 确认 → 发送至少 3 轮工具 + 2 轮等待,max_steps=5(agent_stream.py:528)与 auto-terminate 机制随时可能提前收尾(§3 原因 3)。

## 6. 若要支持「动态组合工具」的改造方向

> 只给方向与涉及模块,不是实施计划。六个方向按依赖顺序排列,前两个是其余一切的地基。

### 方向 A:通用 structured_data 透传协议(去白名单化)
工具返回的 `ToolResult.system`(或新增 `structured` 字段)一律解析为 `{type, payload}` 信封,**无条件**存入 `_tool_structured_results` 并透传到 SSE;按 `type`(而非工具名)路由,解析失败记日志而非静默丢弃。

## 7. 交互僵化点清单(技术实现方便 ≻ 用户体验 的痕迹)

1. **确认卡不可编辑**:邮件主题/留言明明是最该让用户顺手改的字段,卡片却只读(SendEmailConfirmCard.tsx 全文无输入控件)——因为"可编辑"需要结构化回传通道(§4-③),而现状只有"替用户打字"的按钮(CocoChat.tsx:4206-4212)。
2. **按钮=假装打字**:确认发送、%%SUGGESTIONS%% chips(点击即发一句预置话,manus.py:478/998-1002)、IntentChips 的"岗位分析/快速问答"(CocoChat.tsx:5070-5084 只是往输入框填字)——所有交互最终都折叠回"一条文本消息",因为文本是系统唯一的输入通道。
3. **表演性思考**:诊断报告的"思考过程"是 f-string 模板(manus.py:421-428,"正在调用…交叉验证…"),编辑前故意 sleep 450ms 制造"在想"的观感(manus.py:69-71, 1562-1563)。用户看到的努力是演的,真实推理反而被 fast-path 跳过了。
4. **评分不可信**:三个分数由固定算术公式生成(manus.py:363-365),换简历几乎不变;"竞争力=质量+8"这种关系用户多看两份就会穿帮。
5. **同一句话不同命**:问候在前端拦(有 chips)、在后端 fast-path(纯文本)、在 LLM 分类(GREETING_EXCEPTION prompt)三条路径文案与形态都不同,取决于命中哪层词表(#1/#4/#13/#48)。
6. **诊断两阶段靠"接头暗号"**:Phase 2 只认回复里含"先做通用/目标岗位/JD/定向"等词(manus.py:1767-1768);用户自然地说"不用了直接看"会重新走 Phase 1,循环追问。
7. **误伤后无出口**:只读正则误判后 cv_editor 被硬拦并返回错误文案(manus.py:248-257),用户没有"我就是要改"的覆盖手段;同理粘贴长文本必进导入流程(#3),想"就这段文字聊聊"做不到。
8. **组合请求被拆丢**:“优化第一段并翻译成英文"——规则树只有"优化"出口(manus.py:1916-2106),后半句静默蒸发;用户不知道系统只听懂了一半。
9. **卡片挂错/残留的簿记 bug 面**:current→messageId 的 rebind/清理散布 10+ 处且清单互不一致(P0-3),表现为偶发"上一轮的卡片粘在下一轮回答上"。
10. **能力不可发现**:发邮件功能只有 admin 有、且无任何 UI 暗示(工具注册在 manus.py:159-160,入口只有 Composer 角落的邮箱图标);用户不问一句"你能发邮件吗",永远不知道。反之,问了而不是 admin,模型(工具列表里没有)只能凭幻觉回答。
11. **失败静默**:structured_data 任一层漏白名单、type 不匹配即丢(P0-2),用户侧表现为"AI 说完了但什么都没发生",无错误提示。
12. **长对话失忆无提示**:30 条滑窗被裁剪时(chat_history_manager.py:67-78)用户毫无感知,Agent 突然忘记十分钟前给过的 JD,还要用户再贴一遍(session JD 机制只在 OPTIMIZE 链路记一次,manus.py:1920-1925)。
