# Wave 2a + LLM-first 一次性了断 —— 总结问答

- **日期**:2026-07-11
- **分支**:合并至 `dev`(PR #29)
- **背景**:用户问"今天到底优化了什么架构、怎么测试、是否更聪明、是否更接近调研文档描述的理想架构、是否更有灵魂"。本文档整理这次问答,作为这一轮工作的收尾总结。

## 一、今天做了什么

一句话:**把"规则引擎抢在 LLM 前面替它做决定"的代码全删了,让 LLM 自己判断该做什么。**

1. **Wave 2a**(S1-S4c):把堆在 `think()` 里的判定/构造逻辑拆进 `IntentRouter`/`ToolInvocationBuilder`/`ResumeUseCases`/`TurnExecutionState` 几个协作对象,manus.py 从 1578 行降到 794 行。这一步主要是代码整理,还没动"规则 vs LLM"这个核心问题。
2. **LLM-first 一次性了断**(核心):架构审计发现 `AGENT_LLM_FIRST_ROUTING` 默认已经是 `true`,但代码里还留着一整套规则引擎(`resume_use_cases.py` 1172 行、`experience_entry.py` 里"猜用户指哪段经历"的模糊匹配逻辑),这些代码在默认配置下根本不会被执行,是死代码。真删了,连同开关本身——现在没有"退路",Agent 100% 靠 LLM 自己判断。
3. **遗留清理**:`delegation_strategy.py`/`registry.py`/`analyzers/*.py`/`resume_optimizer.py` 孤立死代码簇整体删除。
4. **T1**:收尾几个重构中途的冻结状态(内联单方法类 `ToolInvocationBuilder`、平移让权守卫进 `IntentRouter`)。

manus.py 最终 753 行(从最初 1578 行,减了一半多)。全程委托 opus 子 agent 逐轮独立 review + pytest baseline 对比 + 真实 SSE 链路验证,不是单纯删代码。

## 二、怎么测试

别看代码,直接跟它聊。今天验证过的场景,可以照着复现:

- 说"帮我诊断一下这份简历,目标岗位是 XX" → 之前是硬编码两阶段(先问岗位,回答了才诊断),现在应该一步到位直接给报告
- 故意给一段**没有量化数据**的工作经历,说"帮我改得更有说服力" → 之前可能硬编造数字往上套,现在应该反问要真实数据,不瞎编
- 什么都不给,直接说"帮我生成一份简历" → 同样应该反问要基本信息,而不是编一份假的
- 说"把简历发给 xxx@qq.com" → 应该走 `send_resume_email` 工具,不应该被规则拦住或答非所问

## 三、是否更"聪明"了

有实测证据,不是空话。今天测这几条链路时观察到的真实行为:诊断场景 LLM 自己组织问题去调用分析工具、编辑和生成场景两次都主动拒绝编造数据反问用户、发送邮件场景确认了"规则弃权→LLM 自主编排→正确选中工具"这条链路完整。这些都是真实 SSE 事件日志里看到的。

## 四、是否更接近调研文档描述的理想架构

调研报告(`2026-07-11-agent-architecture-audit-and-paradigm-research.md`)给的判据是"规则只该管安全/防呆边界,理解意图的活该完全交给 LLM"。今天做的正是这件事——之前是"方向选对了但代码没兑现",现在是真的兑现了。这条判据上,现在基本达标。

## 五、是否更"有灵魂"

要诚实说清楚:今天观察到的"拒绝编造、主动追问"这些好行为,根源是删掉规则引擎后,原本被规则代码盖住的 system prompt 里已有的价值观(反编造、防止编出假数据这类)重新生效了——**不是今天刻意给它加了新人设**。

调研报告提到的"给 coco 一套正向价值观核心块"(把散落的否定式禁令收敛成人设,即方案里的 T2)**还没做**,而且方案明确说这个改动风险高,必须先走 brainstorming 对齐,不能直接派 agent 改。所以严格说:今天是"拆掉了压制人设发挥的枷锁",T2 才是"真正主动塑造灵魂"的那一步,还在等用户决定要不要做。
