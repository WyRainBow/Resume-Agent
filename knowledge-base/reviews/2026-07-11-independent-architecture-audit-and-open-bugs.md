### P1 — D5(`Current directory` 矛盾字段)确认仍在,且和新写的价值观正面冲突

`prompt/manus.py:274-275` 仍是:
```
Current directory: {directory}
Current state: {context}
```
`prompt_builder.py:62-64` 把 `{directory}` 填成真实的 `config.workspace_root` 路径。而同一份 prompt 开头价值观第 4 条刚写"用户的世界里没有目录、路径、文件——你的话里也不该有"。**先告诉 LLM 世界上没有目录,又在结尾塞一个真实目录给它**,是自相矛盾,存在潜在泄漏面。工作量小:删两行+重录 golden。

### P2(今天最该被抓但没抓到的一处残留)— 一处"伪装成防呆守卫"的并行关键词判断系统

`conversation_state.py:36-62` 的 `is_read_only_query`/`is_add_experience_query`(`_READ_ONLY_QUERY_RE`/`_WRITE_QUERY_RE` 两张关键词表)是一套**独立于 IntentRouter/LLM-first 运行的并行关键词意图系统**,硬门控两处:
- `manus.py:182-192` `execute_tool` 里**硬拦截** cv_editor_agent(只读轮次直接返回错误,不给 LLM 执行机会)
- `prompt_builder.py:86-105` 注入只读/新增经历的强约束段

这结构上就是"两张判断表互相打架"的同一类反模式,只是披了"防呆"外衣躲过了今天几轮清理。只读拦截本身作为防呆边界可以由规则管(判据允许),但**实现是脆的关键词表,能实测造出假阳性**:

> `is_read_only_query("看看能不能帮我把这段改得更好")` → 写词表里有"改成/改为/改写/润色"但没有"改得",读词表命中"看看" → 判成只读 → `execute_tool` 硬拦截 cv_editor_agent → **用户的编辑请求被静默拒绝**

这是今天这轮认为最该继续动的一处残留规则堆砌。工作量中等,需要一个设计决策(让 LLM 判读写 / 把硬拦截降级成软提示),建议先补一个类似假阳性的回归用例把风险钉住。

## 二、待记录的产品 bug(用户反馈,2026-07-11,暂不排期实施)

### Bug 1:「对话创建简历」流程——选择面板闪现即消失 + 追问字段设计过宽

**现象(用户截图,`om_x100b6a1059c950a0b30037e5e04b773` 附图)**:
用户点"我要优化简历"后,Agent 先列出一份包含**教育经历/实习经历/项目经历/个人经历/自我评价**五类信息的追问清单,引导用户"一次性全发/一段段拆开/先说一项"。接着又触发一轮"先看看你手头有没有现成的简历,我帮你打开选择面板〜"→"看起来你手上还没有简历加载进来〜"这样的对话,**选择面板本身在截图里没有稳定展示出来,像是弹出后立刻判定"没有简历"就消失、转成纯文字追问**——用户描述为"弹出一下就消失了"。

**用户的产品诉求**:
> 做简历时,对于目标院校/教育经历、姓名、年龄、地点、个人简介这些,不用强制要求用户输入。最多要求填"求职意向"就够了。大部分时候要优化的就是几类:实习经历、工作经历、项目经历、专业技能。

即:当前追问清单(教育经历/实习经历/项目经历/个人经历/自我评价五类)覆盖面过宽,应该收窄——核心只应该要**求职意向**,主要经历类信息聚焦在**实习/工作经历、项目经历、专业技能**,不该把教育经历、姓名年龄地点、个人简介这类信息当作强制/默认引导项。

**用户明确要求**:这次只记录,不要求立即修复("你记录一下就行了")。

**涉及文件(初步定位,未深入排查,供后续参考)**:
- 追问清单文案:`backend/agent/prompt/manus.py`"从零生成简历"流程相关段落
- 选择面板闪现行为:前端简历选择面板组件 + Agent 判断"是否已有简历加载"的时序逻辑(具体文件待后续排查)
