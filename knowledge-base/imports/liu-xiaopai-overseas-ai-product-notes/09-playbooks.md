# 09 任务 Playbook

## Playbook 1 判断一个想法是不是值得做

适用场景：
你有一个 AI 产品想法，但不确定它是不是伪需求。

步骤：

1. 写出目标人群，必须具体到一类人。
2. 写出他们在哪个场景下遇到什么麻烦。
3. 判断这个麻烦是否每周或每月反复出现。
4. 找 3 个现有替代方案，看它们哪里差。
5. 问自己：用户不解决会不会有明显损失？
6. 问自己：用户愿意付多少钱让这个麻烦消失？
7. 找真实用户验证，不要只问朋友“觉得好不好”。

输出：

```md
目标人群：
高频场景：
具体问题：
现有方案：
我的差异化：
收费方式：
验证方式：
```

## Playbook 2 写一个能买时间的 Landing Page

适用场景：
你已经知道产品做什么，但首页表达很散。

步骤：

1. 写一句“给谁解决什么问题”。
2. 删除首屏中不能帮助用户理解产品的内容。
3. 准备一个视觉展示，只展示功能理解所需的信息。
4. 设计一个主 CTA，例如 Try it、Get Started、Start Free。
5. 让下方内容露出一点，鼓励用户继续滑。
6. 在第二屏解释关键功能和结果。
7. 在后续区域补充价格、FAQ、案例或社会证明。

输出：

```md
首屏主张：
副标题：
主 CTA：
核心视觉：
下一屏内容：
付费引导：
```

## Playbook 3 接入订阅支付

适用场景：
产品已有登录和基础功能，要开始收费。

步骤：

1. 选择支付平台：Creem、Dodo、Paddle、Stripe。
2. 开启测试模式。
3. 创建产品和价格。
4. 复制 API Key、Product ID、Webhook Secret。
5. 写入 `.env.local` 和部署环境变量。
6. 创建 Checkout 路由。
7. 创建 Webhook 路由。
8. Webhook 验签。
9. 支付成功后更新数据库订阅状态。
10. 前端根据订阅状态展示权益。
11. 用测试卡跑通完整流程。
12. 部署后用公网 URL 重新测试 Webhook。

输出：

```md
支付平台：
测试模式是否开启：
产品 ID：
Webhook URL：
数据库字段：
成功页：
失败页：
验收结果：
```

## Playbook 4 选择 Starter Kit

适用场景：
你要开新项目，但不想从零搭基础设施。

步骤：

1. 明确目标：最快 MVP、学习架构、长期大项目。
2. 列出必须能力：登录、支付、积分、多语言、邮件、后台。
3. 检查支付平台是否支持中国开发者可用方案。
4. 看代码栈是否匹配你的能力。
5. 看是否容易让 AI 读懂和修改。
6. 看文档和 demo 是否完整。
7. 用最小项目跑一次本地启动。

选择建议：

- 快速验证：ShipAny 或课程配套 Starter。
- 学习架构：MkSaaS、MakerKit、SupaStarter。
- 长期复杂项目：优先看架构、权限、数据库和可扩展性。

## Playbook 5 让 AI 帮你开发一个功能

适用场景：
你已经有项目，希望 Codex / Claude Code / Cursor 帮你实现功能。

步骤：

1. 让 AI 先读项目结构。
2. 明确用户角色和权限。
3. 明确要改哪些页面/API/数据库。
4. 要求 AI 先出计划，不要马上写代码。
5. 分小步实现。
6. 每一步跑本地验证。
7. 用真实用户路径验收。
8. 记录遗留问题。

## Playbook 6 上线前最后一轮检查

适用场景：
产品已经能跑，要决定能不能发给真实用户。

步骤：

1. 匿名用户路径。
2. 登录用户路径。
3. 免费额度用尽路径。
4. 付费路径。
5. Webhook 开通权益路径。
6. 支付失败路径。
7. 刷新页面后状态是否保留。
8. 移动端显示。
9. Vercel 日志和错误处理。
10. 敏感密钥检查。
