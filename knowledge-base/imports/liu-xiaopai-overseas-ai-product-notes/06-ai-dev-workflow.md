# 06 AI 编程实操流程

来源页码：Page 051-060, Page 071-080, Page 087-095

## 工具定位

课程中出现的 AI 编程工具：

- Cursor：适合编辑器内协作、改页面、读代码、做局部功能。
- Claude Code：适合终端类项目理解、复杂任务拆解、持续调试。
- Codex：适合终端类代码任务、项目梳理、执行计划、修改和验证。
- V0：适合快速生成 UI 或页面初版。

课程倾向是：复杂任务不要只靠聊天式工具，一定要能读项目、跑命令、改代码、验证结果。

## AI 编程的核心原则

AI 编程不是“让 AI 一口气完成产品”，而是把 AI 当成一个会读代码、会执行命令、会改文件的协作者。

你要负责：

- 定义产品目标。
- 定义用户分层。
- 定义验收标准。
- 控制任务范围。
- 让 AI 分步执行。
- 检查真实结果。

AI 负责：

- 理解现有代码。
- 找相关文件。
- 拆技术计划。
- 写代码。
- 跑命令。
- 根据错误继续修。

如果你不给验收标准，AI 很容易“代码写了很多，但产品路径没通”。

## 先理解项目

Claude Code 可以用 `/init` 生成 `CLAUDE.md`。

Codex 可以用 `/init` 生成 `AGENTS.md`。

目的：

- 梳理项目结构。
- 理解路由、组件、数据库、认证、支付、AI provider。
- 知道核心配置在哪里。
- 后续让 AI 修改时有项目上下文。

## 项目理解要包含什么

让 AI 理解项目时，至少要搞清：

- 页面路由在哪里。
- API 路由在哪里。
- 数据库 schema 在哪里。
- 认证/session 怎么读。
- 支付 provider 怎么封装。
- i18n 文案在哪里。
- UI 组件库是什么。
- 环境变量在哪里定义。
- 本地启动命令是什么。
- 构建/类型检查命令是什么。

这一步越清楚，后面越少乱改。

## 实操项目：AI 换衣服

MVP 设定：

- 产品：AI 换衣服。
- 流量：社交媒体引流。
- 免费版：有水印，每天有限次数，只能单件衣服。
- 付费版：无水印，更高次数，可以同时试穿多件衣服。

用户权限示例：

| 用户类型 | 次数限制 | 水印 | 单次试穿衣服数量 |
| --- | --- | --- | --- |
| 匿名用户 | 不允许使用 | 无意义 | 不允许 |
| 免费登录用户 | 3 次/天 | 有 | 1 |
| 付费用户 | 200 次/月 | 无 | 3 |

核心能力：
使用火山引擎 Seedream 系列模型，多图输入单图输出。图片可以用 base64 方式传输。

## 把产品需求翻译成工程任务

以 AI 换衣服为例，产品需求可以拆成：

前端：

- 上传模特图。
- 上传衣服图。
- 展示生成按钮。
- 展示生成中状态。
- 展示结果图。
- 免费用户结果加水印。
- 付费用户可下载无水印图。

后端：

- 检查用户 session。
- 检查用户额度。
- 调用 Seedream API。
- 保存生成历史。
- 扣减额度。
- 返回结果。

数据：

- 用户表记录 plan 和额度。
- 生成历史表记录输入、输出、状态。
- 积分流水表记录消耗。

权限：

- 匿名用户不能生成。
- 免费用户每天 3 次。
- 付费用户每月 200 次。
- 免费用户单次只能试 1 件衣服。
- 付费用户可试 3 件衣服。

## 开发顺序

1. 明确 MVP 需求。
2. 明确免费/付费权限差异。
3. 找到 API 文档。
4. 让 AI 阅读项目结构。
5. 配置数据库、登录、支付、AI API Key。
6. 让 AI 先出计划，而不是直接写代码。
7. 分模块实现：上传、生成、结果页、水印、额度扣减、付费墙。
8. 跑本地开发服务器。
9. 用真实流程验收。
10. 根据验收结果微调。

## 推荐任务拆法

不要给 AI 一个巨大任务：“帮我做完整 AI 换衣服产品”。更稳的拆法：

1. 先只改 Landing Page 文案和主题。
2. 再做上传表单，不接真实 API。
3. 再接入真实 AI API。
4. 再加登录限制。
5. 再加免费额度。
6. 再加水印。
7. 再加支付后的权限解锁。
8. 再加生成历史。
9. 最后做 UI 打磨和多语言。

每一步都有可验证结果，失败时也容易定位。

## 给 AI 的任务表达方式

好的提示应该包含：

- 背景：我做什么产品。
- 用户分层：匿名、免费、付费分别能做什么。
- 付费规则：额度、套餐、积分、权限。
- 已完成配置：例如 Creem、Webhook、API Key、product id。
- 数据库需求：需要哪些表和字段。
- UI 偏好：价格表位置、主题色、付费引导。
- 执行约束：不清楚先问我，沟通好再编码。

## 任务说明的反例和正例

差的提示：

```text
帮我做一个 AI 换衣服网站，能收费，页面好看点。
```

问题：

- 没有人群。
- 没有权限。
- 没有付费规则。
- 没有验收标准。
- 没有说明现有项目结构。

好的提示：

```text
我正在基于现有 Next.js Starter 做 AI 换衣服产品。
匿名用户不能生成，只能看 Landing Page。
免费登录用户每天 3 次，结果有水印，只能上传 1 件衣服。
付费用户每月 200 次，无水印，可上传 1-3 件衣服。

请先阅读项目结构，找到认证、支付、数据库、AI API 相关文件。
先输出计划，不要改代码。
计划里说明要改哪些文件、加哪些字段、如何验收。
```

## 验收方式

不要只看 AI 说完成。

应该检查：

- 本地服务是否能启动。
- 页面是否能访问。
- 匿名用户是否被引导登录。
- 免费用户是否有次数限制和付费引导。
- 付费用户权限是否打开。
- 支付成功后数据库状态是否更新。
- 多语言文案是否同步。
- 主题色和产品文案是否符合定位。

## 验收应该沿真实用户路径走

验收路径示例：

1. 新用户打开首页。
2. 点击 Try it。
3. 被要求 Google 登录。
4. 登录后进入生成页。
5. 上传图片并生成。
6. 看到免费额度减少。
7. 达到限制后看到付费引导。
8. 点击付费。
9. 支付成功。
10. Webhook 更新权益。
11. 刷新页面后变成付费状态。
12. 生成无水印结果。

这条链路走不通，就不能说产品完成。

## 常见 AI 开发风险

- AI 改了错误的文案文件，导致某语言版本没变。
- AI 只改前端，没有改后端权限，用户可以绕过限制。
- AI 只看成功页，没有实现 Webhook。
- AI 用了不存在的环境变量。
- AI 新增字段但没有迁移数据库。
- AI 没有跑构建，类型错误留到部署才发现。
- AI 把测试 Key 或敏感信息写进代码。

## 我的提炼

AI 编程不是“把需求丢给 AI 等结果”，而是一个管理流程：先让 AI 理解项目，再让 AI 出计划，再分步实现，最后用真实用户路径验收。
