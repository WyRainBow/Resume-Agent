## 1. 操作性定义与分析框架

**灵魂感（本篇口径，用户给定）**：Agent 能主动理解用户意图、灵活组合工具、有记忆和上下文延续、交互自然不做作——而不是每个功能都是写死的固定流程。

每个入选产品按**四问**拆解：

1. **工具怎么被选择 / 组合**：纯 LLM function-calling 自主决策？planner-executor 分层？workflow 图？ReAct 循环 + 计划外化？工具粒度多大？
2. **中间产物怎么传递**：上一步产出（文案 / 文件 / 数据）怎么进入下一步？有没有「工作区 / 上下文对象」？
3. **人机确认怎么做**：破坏性 / 外发动作的确认交互长什么样？文字问答还是结构化卡片？用户能否在确认前**编辑中间产物**？
4. **为什么感觉有 / 没灵魂**：具体机制归因。

**先给全篇结论**（细节见后文）：灵魂感 ≈ **模型的决策自由度 × 过程可见性 × 用户可介入性**。业界所有「有灵魂」的产品都在做同三件事：把编排交还给模型循环（而不是写死流程）、把计划和中间产物外化成用户看得见改得动的对象、把确认下沉到协议层（而不是 prompt 约定）。

## 2. 通用 Agent

### 2.1 Manus（中国团队，通用 Agent 的机制标杆）

Manus 联创 Yichao "Peak" Ji 的工程博客《Context Engineering for AI Agents: Lessons from Building Manus》是目前公开资料里机制密度最高的一手来源（https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus ）。

- **工具选择 / 组合**：单循环 ReAct——每次迭代模型基于当前上下文从**预定义 action 空间**选一个动作，在虚拟机沙箱执行得到 observation，追加进上下文进入下一轮；一个典型任务平均 **~50 次工具调用**，input:output token 比约 **100:1**。三个反直觉设计：
  - **工具不动态增删，只做 logit mask**：动态加载工具会打爆 KV-cache 且让历史消息引用悬空；Manus 用「上下文感知状态机」在解码时屏蔽/强制特定 action（利用 function calling 的 Auto/Required/Specified 三档），工具名用一致前缀（`browser_*`、`shell_*`）便于按状态约束子集。
  - **KV-cache 命中率是第一指标**（缓存与未缓存成本差 10 倍），所以 prompt 前缀稳定、上下文只追加。
  - **错误留在上下文**：失败 action 和 stack trace 不清洗掉，让模型隐式更新先验、避免重复犯错——「错误恢复是真正 agentic 行为最明确的指标」。
- **中间产物传递**：**文件系统即上下文**——沙箱文件系统被当作「无限大、天然持久、可直接操作的外部记忆」，网页可以从上下文丢弃只留 URL、文档内容可省略只留路径（压缩必须可恢复）。所有中间产物（调研笔记、代码、数据）都是沙箱里的文件，下一步工具直接读文件。
- **计划外化（todo.md 复述机制）**：Manus 在任务开始时写 `todo.md` 并**每步重写、逐项打勾**——这不是卖萌，是把全局计划不断复述到上下文**末尾**（模型最近注意力区），对抗 50 步长循环里的 lost-in-the-middle 和目标漂移。2025 下半年架构演进为 planner 子代理返回结构化 Plan 对象 + executor 子代理按 schema 提交结果（约束解码保证格式），并配 knowledge manager 决定什么进文件系统（第三方分析：https://rlancemartin.github.io/2025/10/15/manus/ ）。
- **人机确认**：敏感操作（登录凭据、验证码、付款确认弹窗）**转移控制权给用户**——Browser Operator 模式下用户显式授权、可随时接管键鼠、可随时停（https://manus.im/docs/zh-cn/integrations/manus-browser-operator ）。
- **灵魂感归因**：✅ 计划可见（todo.md 就显示在工作区里，用户看得到它在想什么）＋ 工作区透明（每个文件可点开）＋ 错误自愈。它的「灵魂」不来自人格，而来自**过程像一个真实工作的人**：列清单、犯错、改正、交付。

### 2.2 OpenAI：Operator / Deep Research / ChatGPT Agent / Responses API / AgentKit

OpenAI 的产品线展示了同一家公司内「模型自主循环」与「可视化 workflow」两条路线并存，且最终把确认做成了**产品级交互**而非 prompt 约定。

- **工具选择 / 组合**：
  - **Operator**（2025-01）：CUA（Computer-Using Agent）模型在远程浏览器里以截图→动作循环操作网页，工具粒度 = 原子 GUI 动作（点击 / 输入 / 滚动）（https://openai.com/index/computer-using-agent/ ）。
  - **ChatGPT Agent**（2025-07）：给 ChatGPT 一台**自己的虚拟计算机**，统一工具集 = 文本浏览器 + 可视浏览器 + 终端 + 直连 API/connectors，模型在一个循环里自选工具并可中途被打断改道（https://openai.com/index/introducing-chatgpt-agent/ ；https://venturebeat.com/ai/openai-unveils-chatgpt-agent-that-gives-chatgpt-its-own-computer ）。
  - **Responses API**：把「服务端工具循环」做成 API 原语——内置 web search / file search / computer use / code interpreter + 远程 MCP，对话状态可持久回放（https://openai.com/index/new-tools-for-building-agents/ ）。
  - **AgentKit / Agent Builder**（2025-10）：可视化节点画布（Agent 节点、MCP 节点、Guardrails 节点、**User Approval 节点**），面向企业流程（https://openai.com/index/introducing-agentkit/ ）。注：有第三方报道称 Agent Builder 已宣布弃用、计划 2026-11-30 关停（https://team400.ai/blog/2026-03-openai-agent-builder-visual-workflows ，采用前建议二次确认）——若属实，说明 OpenAI 自己也在从「画布编排」收敛回「模型自主 + 协议层护栏」。
- **中间产物传递**：ChatGPT Agent 的虚拟机是天然工作区——下载的文件、生成的表格 / PPT 存在虚拟机磁盘里供后续步骤使用；Responses API 用 conversation state 串联多轮工具产物。
- **人机确认**（这是 OpenAI 做得最成体系的部分）：
  - ChatGPT Agent 对**高影响动作**（发邮件、下单）执行前必须请求用户确认；对邮件等敏感站点强制 **watch mode**（用户必须盯着）；**takeover mode** 下用户亲自输密码、期间不采集屏幕数据（https://help.openai.com/en/articles/11752874-chatgpt-agent ）。
  - Operator 官方措辞：在产生外部副作用的动作（提交订单、发送邮件）**final 化之前**请求确认（https://openai.com/index/introducing-operator/ ）。
  - 购物场景做成了结构化确认卡：Instant Checkout 里用户逐步确认商品、地址、支付再成交（https://openai.com/index/buy-it-in-chatgpt/ ）。
  - **Agents SDK 的 `needsApproval`**（开发者侧协议层确认，详见 §5.3）。
- **灵魂感归因**：✅ Deep Research 开场先反问澄清（§5.1）；✅ 确认粒度分层（普通动作不问、副作用动作必问、敏感站点盯屏）让人「敢托付」。❌ Operator 早期每步截图慢吞吞、频繁要求接管，被评「像看实习生用鼠标」——**纯 GUI 原子动作粒度太细，反而没灵魂**；ChatGPT Agent 合并文本浏览器（快路径）与可视浏览器（兜底路径）后体验才顺。

### 2.3 Anthropic：Claude Code / Agent SDK / MCP / Skills

Anthropic 的两篇工程文章几乎就是「灵魂感机制」教科书：《Building Effective Agents》（2024-12，https://www.anthropic.com/research/building-effective-agents ）与《Building agents with the Claude Agent SDK》（2025-09，https://www.anthropic.com/engineering/building-agents-with-the-claude-agent-sdk ）。

- **工具选择 / 组合**：
  - 总纲：区分 **workflow**（LLM 和工具走预定义代码路径）与 **agent**（LLM 动态指挥自己的过程和工具使用）；能用简单模式就不要上 agent；五种可组合 workflow 模式（prompt chaining / routing / parallelization / **orchestrator-workers** / evaluator-optimizer）。
  - Agent SDK 的循环：**gather context → take action → verify work → repeat**。工具以字符串名声明（`allowedTools`），SDK 托管循环、自动上下文压缩；只读工具可在一个 turn 内并行执行。
  - **Subagents**：orchestrator 按任务描述派生子代理，各自隔离上下文窗口、只回传相关结论——既并行又防上下文污染。
  - **工具粒度设计观**：「工具在上下文窗口中占显著位置，是模型优先考虑的动作」——工具应对应**高频主干动作**（如 `searchEmails`），长尾灵活操作交给 bash / **代码生成**（「Code is precise, composable, and infinitely reusable」——生成 Excel/PPT 之类复杂产物直接写代码），第三方集成走 MCP。
  - **Skills 渐进披露**（2025-10，https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills ）：技能 = 一个装着 SKILL.md 的目录，**三层加载**——YAML frontmatter 常驻 system prompt（只花几十 token 让模型知道何时用）→ 命中才读 SKILL.md 正文 → 更深的 reference 文件按需读。**新能力 = 写文档，不是写代码**。
- **中间产物传递**：文件系统 =「可能被拉进上下文的信息」；agent 用 grep/tail 等 bash 决定加载什么（agentic search）；官方博客通篇用**邮件 Agent** 做例子：历史邮件存 `Conversations/` 文件夹供搜索、草稿写成文件、生成 Python 处理 PDF 附件。
- **人机确认**：多层协议化——
  - **权限系统**：每个工具调用先过 permission 规则 / `canUseTool` 回调，命中即挂起询问用户（allow once / always / deny），这是**运行时拦截**而非模型自觉（https://code.claude.com/docs/en/agent-sdk/user-input ）。
  - **AskUserQuestion 工具**（§5.2）：模型主动发结构化选择题。
  - **MCP elicitation**（§5.4）：服务器在工具执行中途按 JSON schema 向用户要结构化输入。
- **验证闭环**（verify work 三式）：规则反馈（lint 类确定性校验）→ 视觉反馈（**截图生成的 HTML 邮件**检查排版）→ LLM as judge（子代理评审草稿语气是否像用户本人）。
- **灵魂感归因**：✅ 这套体系的灵魂感来自「**该问就问、该做就做、做完自查**」：AskUserQuestion 前置澄清 + 权限系统兜底确认 + verify work 自我把关。❌ 代价是终端 UI 对非开发者不友好——灵魂感机制和大众可用性是两回事。

### 2.4 Devin（Cognition）

- **工具选择 / 组合**：planner-executor 风格——接任务先分析代码库、产出结构化计划（读哪些文件、改什么、跑什么测试），然后 plan → execute → observe → **re-plan** 循环；官方定位「初始计划是起点不是合同」，执行中持续修订（https://docs.devin.ai/get-started/first-run ；https://medium.com/@nitinmatani22/how-devin-ai-actually-thinks-autonomous-planning-dag-execution-and-dynamic-re-planning-explained-997be175a475 ）。工具 = 沙箱开发环境的 shell + 编辑器 + 浏览器（原子粒度）。
- **中间产物传递**：持久沙箱是工作区——代码、测试输出、浏览器调研结果都留在环境里跨步骤复用。
- **人机确认**：**计划先行确认**——开工前把 plan 发给用户求 yes/no；跑在 Slack 里时 @Devin 在 thread 内回进度、**在 thread 内追问**（https://slack.com/marketplace/A06A3TU8H39-devin ）。
- **记忆分层**：**Knowledge**（全局知识：风格指南 / 架构，所有 session 自动引用）与 **Playbooks**（特定任务的流程手册，按任务挂载）两层分治（https://www.shunku.net/blog/devin-ai-practical-guide/ ）。
- **灵魂感归因**：✅ 「先给你看计划、你点头我才动、做的过程随时汇报、你在哪（Slack）我就在哪」——灵魂感来自**协作礼仪**；Knowledge/Playbooks 让它「记得你们团队的规矩」。❌ 自主时长过长、过程不透明时用户信任骤降（社区常见抱怨是放养后跑偏）。

### 2.5 Genspark / Flowith / Lovart（多 agent 混合编排三兄弟）

**Genspark Super Agent**：
- 编排 = **mixture-of-agents**：中央 orchestrator 拆解请求 → 子任务路由给挂在不同前沿模型（GPT-5.1 / Claude / Gemini 等）上的专家 agent → 80+ 工具执行 → 各模型**互相校验**结果 → 汇编成 Sparkpage 成品（https://www.qed42.com/insights/genspark-super-agent-the-rise-of-next-generation-ai-assistants-and-how-it-compares-to-manus-ai ；https://genspark.cloud/super-agent/ ）。
- 中间产物：以最终 Sparkpage / 文件为中心，过程产物用户不可编辑。
- 灵魂感：❌ 偏「黑盒外包公司」——多模型互验降低幻觉，但用户对过程的介入点少，惊艳但不亲密。

**Flowith（Agent Neo）**：
- 编排 = **无限画布上的多线程**：一次会话可开多条并行线程（调研 / 写作 / 编码……），节点间连线互相引用，Oracle 模式动态重排优先级，宣称无限步骤 / 无限上下文（https://www.oschina.net/news/350795 ；https://36kr.com/p/3311450425188102 ）。
- 中间产物：**每个节点就是中间产物**，可视、可拖拽、可作为其他线程输入——画布本身就是「工作区对象」的 UI 化。
- 灵魂感：✅ 计划与产物全部摊在画布上，「思考结构」可见；❌ 学习成本高，画布自由度大到普通用户不知从何下手——可见性过载同样伤灵魂感。

**Lovart（设计垂直 Agent）**：
- 编排 = 需求拆解成主题 / 风格 / 素材 / 版式四条流水线，「Creative Reasoning Engine」调度多模态模型（图 / 视频 / 音频 / 3D）产出**保留图层的可编辑资产**，可导入 Figma/PS（https://36kr.com/p/3311450425188102 ）。
- 人机确认（它最出圈的机制）——**ChatCanvas 圈选批注**：用户在画布任意位置圈选 / 打 pin 写修改意见，一次可堆几十个批注点后点「Run All」批量执行；关键决策前**主动向用户解释「为什么这么设计」再继续**（https://letschuhai.com/71ca9acf ；https://m.aitntnews.com/newDetail.html?newId=16530 ）。
- 灵魂感：✅ 三点归因——中间产物可编辑（图层不是死图）、确认不是「是/否」而是**在产物上直接批注**、主动解释设计逻辑（拒绝黑盒）。这是「确认前编辑中间产物」交互的天花板。

## 3. 国内平台与模型厂

### 3.1 Coze（扣子）vs Dify：workflow 图路线的启示（反面教材式参照）

- **Coze**：字节零代码 Agent 平台，编排 = **可视化 workflow 节点图**（基础节点 / 插件 / 子工作流三类节点，可单节点调试，https://www.coze.cn/open/docs/guides/workflow_node ）；2026-01 的 Coze 2.0 引入「技能封装」「长期规划」概念、强调多 Agent 协作与长期记忆（https://aiho.net/agent/platform/coze.html ）。
- **Dify**：开源 LLM 应用平台，可视化 workflow 画布（LLM 调用 / 知识检索 / 条件分支 / 代码执行节点）+ 单 Agent 节点，强在节点级执行日志与生产可控性；设计哲学是**单 Agent 可控性优先**（https://www.woshipm.com/ai/6205201.html ；https://www.cnblogs.com/hogwarts/p/19197024 ）。
- 中间产物：节点间以**变量引用**传递（上游节点输出显式映射到下游输入）——数据流是图的边，用户搭建时可见，但终端用户运行时不可介入。
- 人机确认：靠在图里手工插「等待人工」节点，非一等公民。
- **灵魂感归因**：❌ 这条路线是「每个功能都是写死流程」的极端形态——确定性最高、灵魂感最低：意图理解被路由节点锁死、组合方式被连线锁死、遇到图之外的需求就答非所问。**对我们的意义是刻度尺的另一端**：Resume-Agent 的大颗粒工具（如 `send_resume_email` 焊死三步）本质上就是把一张三节点 workflow 图藏进了一个工具的 execute 里。

### 3.2 Kimi（月之暗面）：K2 模型层 + OK Computer 产品层

- **模型层（K2/K2.5）**：为 agentic 能力专门造数据——汇集 **3000+ 真实工具 + ~20000 合成工具**，在模拟 + 真实环境批量生成多轮工具调用轨迹做训练，RL 阶段用可验证奖励（RLVR）+ 自我批评 rubric（K2 技术报告：https://arxiv.org/pdf/2507.20534 ）。启示：工具组合的「灵魂」一部分是**训练出来的**，不全是编排框架给的。
- **产品层（OK Computer agent，逆向分析：https://github.com/dnnyngyen/kimi-agent-internals ）**：
  - 工具编排：四层工具栈——kernel 控制面（FastAPI :8888 管环境生命周期）/ IPython 计算层 / Playwright 隐身 Chrome 网络层 / 文件层；OK Computer 模式 29 个工具 vs 基础聊天 9 个，且**工具调用预算近乎无限**（基础聊天限 10 步）。
  - 中间产物：持久工作区 `/mnt/okcomputer/`（基础聊天只有只读、每轮重置的 `/mnt/kimi/`）——产物（docx/xlsx/网站）全部落盘传递。
  - **技能即文档**：Sheets 模式强制「必读 xlsx SKILL.md」（925 行，含**交付前必须运行的验证命令**）；Slides 模式则换人格 prompt（「你是在麦肯锡干了 20 年的演示设计师」）。「写一本好手册放对目录，agent 就成了专家」——与 Claude Skills 殊途同归。
- **灵魂感归因**：✅ 能力扩展靠文档注入而非代码分支，模型保留组合自由；✅ 交付前跑验证命令（自查）。❌ 验证是生成式的（agent 自己写校验逻辑），错误可能无声传播。

### 3.3 智谱 AutoGLM（沉思 / Phone Use）

- 编排：「**边想边干**」——思考模型（RL 训练出自我批评 / 反思能力）与执行穿插，而非先想完再干；深度调研能力与 GUI 操作能力合体（2025-03 发布，https://finance.sina.com.cn/tech/csj/2025-03-31/doc-inerpqhq7160075.shtml ）。开源的 Open-AutoGLM 用 ADB 驱动安卓真机、多模态读屏（https://github.com/zai-org/Open-AutoGLM ）；社区 GUI 工程提供两种模式，其一是 **planner 层通过工具调用驱动 executor 层**、便于调试与策略迭代（https://github.com/suyiiyii/AutoGLM-GUI ）。
- 灵魂感归因：✅「边想边搜 / 边想边干」的穿插式推理比「一次规划到底」更像人；❌ 手机 GUI 操作的原子动作粒度 + 界面变化脆弱性，仍常见卡壳需人救场。

### 3.4 MiniMax Agent：Leader–Worker–Verifier 三角色 + 对抗式验收

- 编排：分层协作——**总调度 Agent 理解意图拆子任务，执行 Agent 各干各的**（可 7 个并发）（https://www.minimaxi.com/news/minimax-agent ）；Agent Team 进一步把角色收敛为三类：**Leader**（用户目标 → 任务结构）、**Worker**（不同工具与上下文）、**Verifier**（验收：查事实来源、对覆盖清单）（https://zhuanlan.zhihu.com/p/2037877345634276836 ）；Mavis 框架引入 Worker 与 Verifier 的**对抗机制**，用状态机控制角色职责与交接时机，以工程确定性约束模型随机性（https://www.firecat-web.com/daily-news/8474 ）。
- 中间产物：1M 上下文窗口 + Memory 管理机制，长任务（数小时到数天）不失忆。
- 灵魂感归因：✅ **把「验收」建成独立角色**是国产队里最有借鉴价值的一手——产出先过内部质检再见用户，减少「一本正经交垃圾」的掉灵魂瞬间。

### 3.5 字节豆包（简要）

公开机制资料较薄，两点值得记：① 豆包 1.5 深度思考模型的「**边想边搜**」——推理链内穿插多轮搜索（想→搜→再想），工具调用长在思维链里而非外挂（https://www.nbd.com.cn/articles/2025-04-17/3836735.html ）；② 火山引擎 OS Agent 方案（UI-TARS GUI 模型 + 云手机 / 云服务器）走的是 GUI 原子操作路线（https://www.geekpark.net/news/348431 ）。机制细节披露不足，本篇不深展开。

## 4. 垂直参考：coding agent 的交互沉淀

### 4.1 Cursor：Plan Mode 是「计划可编辑」的标准答案

- 编排：agent 模式 = ReAct 循环（读文件 → 多文件编辑 → 跑终端命令 → 迭代到完成）；2.0 起用 RL 训练的自研 Composer 模型 + agent 优先界面，多 agent 并行（各自模型 / 各自 git worktree）（https://www.digitalapplied.com/blog/cursor-2-0-agent-first-architecture-guide ）。
- **Plan Mode 机制**（https://cursor.com/blog/plan-mode ；https://cursor.com/docs/agent/plan-mode ）：进入计划态后模型获得**专门的建计划 / 改计划工具**——先调研代码库、**主动问澄清问题**，产出一份含文件路径、代码引用、Mermaid 图和 TODO 清单的 **Markdown 计划文件**；用户可**直接内联编辑计划**（增删 to-do），满意后一键转执行。
- 中间产物：计划本身是文件（可进版本库）；编辑 diff 逐文件呈现待 accept。
- 人机确认：终端命令白名单外逐条确认；编辑以 accept/reject diff 呈现。
- 灵魂感归因：✅ 「计划是用户可编辑的中间产物」——用户改的不是 prompt，是 Agent 的**待办清单本身**，掌控感极强。

### 4.2 Replit Agent 3：自主性做成旋钮 + REPL 自测

- 编排：主 agent 给 subagent 下发高层计划，subagent 执行后**汇总测试结论**（什么能用什么坏了）再回主 agent（https://blog.replit.com/introducing-agent-3-our-most-autonomous-agent-yet ）。
- **Autonomy Level 四档**：Low（几乎每步过目）→ Medium → High → Max（200 分钟+ 放养、自建任务清单、自我监督）——**自主性是用户可调的旋钮，不是产品定死的常量**（https://docs.replit.com/replitai/autonomy-level ）。
- **REPL-based 自测**：代码执行 + 浏览器自动化像真实用户一样点应用，抓「Potemkin interface」（看着能用实际是摆设的界面），数百步测试中位成本 $0.20/session（https://blog.replit.com/automated-self-testing ）。
- 灵魂感归因：✅ 自主档位让「多久打扰你一次」成为用户选择；✅ 自测让「我做完了」可信。

### 4.3 Perplexity Comet：高自主浏览器 agent 的正反两面

- 编排：常驻浏览器 agent，顺序执行工具直到问题解决；五类工具（web 搜索 / 浏览器操作 / 邮件日历 / 代码解释器 / 记忆管理）；直接操作用户**已登录的真实标签页**（无需 API 对接 Gmail），外加 MCP connectors（逆向分析：https://deepwiki.com/x1xhlol/system-prompts-and-models-of-ai-tools/6.1-comet-browser-assistant-by-perplexity ；https://labs.zenity.io/p/perplexity-comet-a-reversing-story ）。
- 灵魂感归因：✅ 「就在你干活的地方干活」（浏览器上下文天然共享）；❌ 反面教材：L4-L5 高自主 + 确认弱化，被安全研究者演示 prompt injection 可拐走它执行恶意操作（arXiv AI Agent Index 将其列为部署 agent 中自主等级最高一档：https://arxiv.org/pdf/2602.17753 ）——**没有确认层的灵魂感是恐怖谷**。

## 5. 确认交互与「交互灵魂感」标杆（专题）

这一节回答用户最关心的问题：**结构化确认到底长什么样、业界怎么让用户在确认前改中间产物**。

### 5.1 ChatGPT Deep Research：开场澄清（clarify-first）

收到任务后**不立刻开跑**，先反问 2-4 个澄清问题（报告格式？目标读者？要不要数据表？口径范围？）并给出研究计划让用户调整后才出发（https://help.openai.com/en/articles/10500283-deep-research-in-chatgpt ）。机制价值：把「理解偏差」消灭在最贵的长循环开始之前。**澄清是一种前置确认**。

### 5.2 Claude Code AskUserQuestion：结构化提问作为一等工具

Claude Code v2.0.21 引入的**原生工具**：模型对意图不确定时调用它，一次 1-4 个问题、每题 2-4 个选项（可多选、可自由输入），选项带推荐标记和解释，终端渲染成键盘可导航的选择卡（https://code.claude.com/docs/en/agent-sdk/user-input ；https://claudelog.com/faqs/what-is-ask-user-question-tool-in-claude-code/ ）。关键设计：**「问用户」和「查文件」是同级的工具调用**——澄清进了 ReAct 循环本身，模型自己决定何时问，而不是前端写规则拦截。

### 5.3 OpenAI Agents SDK `needsApproval`：确认下沉到运行时协议

工具定义可带 `needsApproval: true`（或按参数动态判定的 async 函数）。触发时**工具调用不执行**，run 挂起并在结果的 `interruptions` 数组里返回待批项；开发者调 `state.approve(interruption)` / `reject()`（可 `alwaysApprove` 记住决定），再把同一个 state 传回 `runner.run()` **原地续跑**（https://openai.github.io/openai-agents-js/guides/human-in-the-loop/ ）。AgentKit 的 Agent Builder 里同类能力是 **User Approval 节点**：「要发送这封邮件吗？」→ 人在面板点 Approve → 流程才进入 Gmail MCP 节点，可配超时升级（https://platform.openai.com/docs/guides/node-reference ）。**本质：确认 = 循环的挂起 / 恢复语义，不是求模型自觉。**

### 5.4 MCP elicitation：服务器中途向用户要结构化输入

MCP 2025-06-18 规范新增 `elicitation/create`：服务器在工具执行**中途**发起请求，附一个受限 JSON Schema（仅原始类型），客户端渲染成表单收集用户输入；支持表单模式和 URL 模式（OAuth 类带外流程）；用户可接受 / 拒绝 / 取消（https://modelcontextprotocol.io/specification/2025-06-18/client/elicitation ）。这把「问用户拿参数」标准化成了**协议能力**，任何客户端都能渲染。

### 5.5 浏览器接管（Operator / Manus）：确认的物理形态

敏感输入不走「问答」而走**控制权移交**：Operator 的 takeover mode 让用户亲手输密码 / 支付信息（期间不截屏）；Manus Browser Operator 遇登录 / 验证码 / 付款弹窗时把键鼠还给用户，做完再交回。**「你来做这一步」是最高等级的确认交互。**

### 5.6 Gemini in Gmail：草稿交接（draft handoff）——「发邮件」的保守解

Help me write 生成的邮件**永远以草稿形态落进 Gmail 原生撰写窗 / Drafts 文件夹**，用户在原生界面里改完自己点发送——AI 从不代发（https://support.google.com/mail/answer/13955415 ）。机制价值：把「确认」变成用户本来就会做的动作（在熟悉界面里点发送），零学习成本；**撰写与发送在产品层被物理分离**。

### 5.7 Linear AIG / Notion Agent：agent 作为「同事」的礼仪规范

Linear 的 Agent Interaction Guidelines 把交互原则写成了对第三方 agent 的**准入规范**（https://linear.app/developers/aig ）：身份必须清晰（永远不可被误认为人）、用平台已有 UI 模式做事（issue / comment / 状态流转，而非私有弹窗）、被调用后立即给不打扰的反馈、内部状态透明（在想 / 在等输入 / 在执行 / 完成）、被请离立即退出、**最终责任永远留给人**。Notion Agent 则在删页等破坏性动作上强制确认、以行内评论 / 建议编辑的形式呈现改动（https://www.notion.com/help/notion-agent ）。**灵魂感的下限是礼仪：不冒充人、不静默、不越权。**

### 6.2 灵魂感五支柱（机制层共性）

1. **编排交给模型循环，流程图只藏在护栏里**。有灵魂的产品无一例外是「LLM 在循环里自选工具」，写死流程的部分（Guardrails、状态机、Verifier）只做约束不做主干（Anthropic「workflow vs agent」分野、Manus logit mask、MiniMax 状态机管交接）。
2. **计划外化：想什么让用户看见，且看得见就该改得动**（Manus todo.md、Cursor Plan Mode 内联编辑、Devin 计划先行确认、Flowith 画布）。
3. **中间产物是一等公民**：统一落在「工作区」（文件系统 / 虚拟机磁盘 / 画布 / 图层 / Drafts 文件夹），既是下一步工具的输入，也是用户编辑的对象（Manus/Kimi 文件系统、Lovart 图层、Gemini 草稿）。
4. **确认下沉到协议层**：`needsApproval` 挂起-恢复、权限回调、elicitation 表单、Approval 节点、浏览器接管——确认是运行时语义，模型「想绕也绕不过」；且确认卡里的内容（邮件草稿、订单明细）**可编辑后再放行**。
5. **澄清前置 + 自验后置**：开工前结构化提问（Deep Research、AskUserQuestion、Cursor Plan Mode），交付前自测（Replit REPL、Agent SDK verify work、MiniMax Verifier、Kimi 验证命令）。

### 6.3 专题：业界怎么做「发邮件 = 撰写 + 确认 + 附件 + 发送」

用户的核心疑问，业界的答案惊人一致——**「撰写」和「发送」永远是两个动作，中间隔着一个可编辑的草稿**：

| 产品 | 链路 |
|---|---|
| Anthropic Agent SDK（官方邮件 agent 示例） | 搜历史邮件（subagent 并行）→ 写草稿成**文件** → 截图验证 HTML 排版 → 子代理评审语气 → 发送工具执行（https://www.anthropic.com/engineering/building-agents-with-the-claude-agent-sdk ） |
| ChatGPT Agent | 起草 → **发送前强制用户确认**（邮件属高影响动作），邮件站点还要 watch mode（https://help.openai.com/en/articles/11752874-chatgpt-agent ） |
| AgentKit（Agent Builder） | Agent 节点起草 → **User Approval 节点**（"要发这封吗？"，面板里 Approve）→ Gmail MCP 节点发送（https://platform.openai.com/docs/guides/node-reference ） |
| OpenAI Agents SDK | `send_email` 工具标 `needsApproval:true` → run 挂起返回 interruption →（应用层可让用户改参数）→ approve → 原 state 续跑（https://openai.github.io/openai-agents-js/guides/human-in-the-loop/ ） |
| Gemini in Gmail | AI 只负责起草，草稿落进原生 Drafts，**人类自己点发送**（https://support.google.com/mail/answer/13955415 ） |

四个不变式：① 草稿是**落盘的中间产物**，不是工具内部变量；② 附件渲染（PDF 等）是**独立步骤 / 独立工具**；③ 确认发生在**副作用工具执行前一刻**，由运行时挂起而非模型自觉；④ 确认界面允许**编辑草稿本身**（改完的版本才是被发送的版本）。

## 7. 对 Resume-Agent 的启示

**我们的现状痛点**（对照基线）：
- P1 大颗粒工具：`send_resume_email` 一个 execute 焊死「渲染 PDF + 固定留言 + SMTP 发送」，用户无法先写 / 改求职信再发；
- P2 新增一个工具要动 ~10 处注册点（工具类、tool/__init__、manus.py 白名单、toolcall.py structured_data 白名单、agent_stream.py SSE 白名单、前端事件 router、专属卡片、时间线 / 流式两处接线、状态 rebind / 清理）；
- P3 意图识别 = 规则（edit_rules）+ LLM 混杂，问候 / 粘贴导入在前端 fast-path 绕过 Agent；
- P4 确认交互靠 prompt 约定（confirm 参数写在工具 description 里求模型遵守）；
- P5 记忆只有会话内 30 条滑窗，无长期记忆。

**八条可借鉴设计**（每条注明出处机制 → 解决的痛点）：

1. **把 `send_resume_email` 拆成原子工具三件套，让循环去组合**（借鉴：Anthropic Agent SDK 邮件示例的「草稿文件 → 验证 → 发送」+ Manus 原子 action space → 解决 P1）。拆成 `write_cover_letter`（产出草稿，落工作区并推前端可编辑卡）、`render_resume_pdf`（已有链路复用）、`send_email(to, subject, body, attachment_refs)`（只管发）。「发简历给 HR」不再是一个函数，而是模型在 ReAct 循环里的一段自然组合——用户中途说「语气改软一点」时，模型只需重跑 `write_cover_letter`，而不是整个流程重来。
2. **确认从 prompt 约定升级为运行时挂起**（借鉴：OpenAI Agents SDK `needsApproval` 的 interruption/approve/resume 语义 + Claude Code `canUseTool` 拦截 → 解决 P4）。给工具加 `requires_approval` 元字段，`toolcall.py` 执行前命中即挂起循环、SSE 推 `approval_request` 事件（含完整参数 + 哪些字段可编辑），前端渲染确认卡，用户批准（可先改参数，如邮件正文）后带修改后的参数恢复执行。模型再「不自觉」也绕不过去。
3. **ResumeDataStore 泛化为会话工作区（named artifacts）**（借鉴：Manus「文件系统即上下文」+ Kimi `/mnt/okcomputer` 持久工作区 + Lovart 可编辑图层 → 解决 P1 的中间产物断层）。工作区 = `{resume, cover_letter, jd_analysis, plan, ...}` 的命名槽位；工具声明自己读 / 写哪些槽位；每次写入推一种 `artifact_updated` 事件，前端把槽位渲染成可编辑卡片，用户编辑写回工作区——下一步工具读到的就是用户改过的版本。这正是「确认前编辑中间产物”的数据基础。
4. **新增 `ask_user` 结构化澄清工具**（借鉴：Claude Code AskUserQuestion 的 1-4 问 × 2-4 选项 + MCP elicitation 的 schema 化表单 + Deep Research 开场澄清 → 解决 P3）。让「问用户」成为与「改简历」同级的工具调用，模型自主决定何时问（「你要投的是后端还是算法岗？」「邮件语气正式还是亲切？」）；前端一个通用问题卡组件渲染选项。有了它，前端 fast-path 规则可以逐步退役——意图歧义交回给模型，用提问而不是猜测消解。
5. **一种自描述事件信封收敛 10 处注册点**（借鉴：MCP 统一工具协议 + Agent SDK 字符串声明 `allowedTools` 的低摩擦注册 → 解决 P2）。`ToolResult.system` 统一为 `{type, payload}` 自描述信封；SSE 层只透传一种 `tool_result` 事件不再逐工具开白名单；前端用 `CardRegistry: Record<type, Component>` 按 type 分发、未知 type 走通用 JSON 卡兜底。新工具的最小改动收敛到 2 处：后端工具类 + （可选）前端专属卡片。
6. **多步任务计划外化 + 复述**（借鉴：Manus todo.md recitation + Cursor Plan Mode 可编辑计划 + Devin 开工前计划确认 → 解决长任务漂移与过程黑盒，兼治 P5 的滑窗遗忘）。「按 JD 改简历并发给 HR」类任务先产出 checklist 计划（SSE 推前端展示、用户可增删项），每步完成打勾，计划文本每轮追加在上下文尾部——30 条滑窗里丢的目标，靠复述留在注意力窗内。
7. **记忆分层：会话滑窗之外加用户级 Knowledge**（借鉴：Devin 的 Knowledge（全局自动引用）/ Playbooks（任务流程按需挂载）二分 + Claude Skills 渐进披露的加载哲学 + 前篇 ColaOS「质变触发」的克制写入 → 解决 P5）。用户级记忆存求职方向、目标岗位、语气偏好、常用称呼（低频更新、每会话注入 system）；「投递流程」类固定套路写成 playbook 文档按需加载，而不是焊进代码。
8. **错误留在上下文 + 交付前自验**（借鉴：Manus「keep the wrong stuff in」+ Replit REPL 自测抓 Potemkin interface + Agent SDK verify work 三式 + MiniMax Verifier 角色 → 提升「说完成时真的完成」的可信度）。工具失败的 traceback 原样作为 observation 回给模型让它改参重试而非吞掉；`show_resume` / 发送前跑规则校验（PDF 页数、空 section、附件存在性），校验不过不许声称完成。

**优先级建议**：2（确认协议层）+ 3（工作区）+ 1（拆工具）是一组，共同兑现「发邮件 = 撰写 + 确认 + 附件 + 发送」；4（ask_user）与 5（事件信封）次之，分别治意图与扩展性；6/7/8 随后迭代。
