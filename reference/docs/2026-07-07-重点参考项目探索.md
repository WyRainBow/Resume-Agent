## 一、Resume-Matcher：模板简历怎么来的

**核心答案：7 个模板全部是自研 React 组件 + CSS Module，没有任何 LaTeX**。模板在 `apps/frontend/components/resume/`（每模板一个 tsx + 一个 css module），注册表在 `lib/types/template-settings.ts`。

### 用户觉得「像我们默认模板」的就是它的 `latex` 模板——判断成立

我们的默认模板是 billryan/resume 系（`latex-resume-template/resume.cls`，经 hijiangtao 分叉修改）。对照它的 `resume-latex.tsx`：

| 视觉特征 | 我们（真 LaTeX） | 它（CSS 复刻） |
|---|---|---|
| 姓名 | `\Huge\scshape` 居中小型大写 | `small-caps` 居中加粗 serif |
| 联系行 | 居中 · 分隔 | 居中 · 分隔 |
| 分节标题 | `\titlerule` 通栏横线 | `border-bottom` 通栏横线 |
| 条目行 | 粗体左 + 日期右 | 粗体左 + 日期右 |
| 字体/配色 | Times 系 serif 全篇黑白 | 全 serif 黑白 |

两者是**同一个「经典 LaTeX 简历」视觉基因**：它代码注释自述 *"reminiscent of the popular LaTeX résumé templates"*——是对这一门类（Jake's Resume / billryan 系）的泛化致敬，非移植。另一个 `vivid` 模板注释明确写 "Awesome-CV lineage"。其余：swiss-single/swiss-two-column（自家瑞士设计系统）、modern/modern-two-column（accent 色）、clean（极简 sans）。

### 模板机制与 PDF 链路

- 数据/样式彻底分离：单一 ResumeData + `sectionMeta` 排序，模板 = `(ResumeData, settings) → JSX` 纯渲染函数；17 个排版参数（页距/间距/字号/字体/主题色）经 `settingsToCssVars()` 变 CSS 变量，`_base.module.css` 统一消费。**加一个模板 = 1 个 tsx + 26~155 行 css**。
- PDF：后端 Playwright 无头 Chromium 访问前端打印页（React Server Component）→ `page.pdf()`；**预览与导出同一渲染源**，天然 WYSIWYG。
- 浏览器预览分页是另一套 DOM 测量（`use-pagination.ts`），与 Chromium 真实分页可能漂移。

### 对我们做「多模板」的启示

**学**：①数据/样式分离 + 单一 registry（它注册点分散 6 处靠注释同步，是它自己的坑，我们要收敛成一个 map）；②模板注册守卫测试（断言 id/元数据齐全，治我们 CLAUDE.md「影响范围枚举」痛点）；③每模板签名字体预设 + 用户可覆盖。
**避**：HTML→Chromium 中文排版控制弱于 XeLaTeX（字距/断行/字体嵌入），且引入 Chromium 重依赖 + 后端须能访问前端打印页的部署耦合。**结论：我们做多模板应留在 LaTeX 层（多 `.cls` / 多组生成风格），不开 HTML 第二渲染链路**。

## 二、vibe-resume：理念 demo，两件事值得抄

**现实校准**：总计约 1000 行的静态模板仓库（1 HTML + 1 CSS + 130 行导出脚本 + 1 个 SKILL.md），无框架/后端/数据库/AI SDK。理念：**HTML 当简历源文件、AI coding agent 当编辑器、PDF 当构建产物、Git 管版本**。

**值得抄的两件**：
1. **Chromium screen-media 动态测高导出**（`scripts/export-pdf.mjs`，130 行）：`emulateMedia('screen')` 绕开 print 分页差异 → 等字体就绪后 JS 测量内容实际高度 → 自定义 `@page size` 输出**内容多高页面就多高的单页 PDF**。已实测产物为单页变高（非 A4 分页）。若未来要「网页预览与 PDF 严格一致」的轻量导出选项，这是被验证的最短路径。
2. **「AI 编辑契约 + 改后机器验证」**（`skills/vibe-resume-editor/SKILL.md`）：编辑纪律（不编造学历/雇主/数据、强动词+可度量结果、内容与样式分文件改）+ **改完必须跑导出并用 pdfinfo/pdftotext 断言页数与 section 完整**。可移植进我们 cv_editor_agent / generate_resume 的工具链：AI 改完简历后自动做产物级断言，而不是只信 LLM 说改好了。另有防幻觉细则：mock 内容必须可见标注「（仅为示例演示）」。

**别学**：HTML 当数据源（无结构化数据→没法做我们的表单/多简历/评分）；无 diff/撤销（我们的 resume_patch 链路是代差领先）；导出脚本硬编码作者本机 Chromium 路径、中文字体依赖本机系统字体。

## 三、shushu-internship-tool：不是投递管理，是「JD→可面试项目」的 Skill 包

**纠偏**：没有投递记录/公司/进度实体，无数据库无 UI。它是面向 Codex/Claude 的 **AI Skill 包**（247 行 SKILL.md 工作流 + 3 个零依赖 Python 脚本），解决「JD → 能投递、能面试、能讲清的项目素材」，恰好是求职闭环里我们没覆盖的上游。8 步工作流：JD intake（含一次性结构化偏好 gate）→ 找 2-3 个 GitHub 候选项目 → `candidate_score` 打分排序 → `repo_audit` 审计 → 分档运行（interview-only/smoke/full）→ 魔改建点 → `interview_pack` 面试包 → 投递就绪 gate。

**与「低频→高频」的关联（最重要收获）**：
1. **「JD 一等实体 → 一 JD 一简历版本 → 投递前 readiness 检查表」的闭环形态**：版本列表按目标岗位/公司组织而非「副本1/副本2」；每个 JD 版本挂一张「能投了吗」卡（JD 命中度、量化口径统一、PDF 已导出、Q&A 已备）。这是把投递从「导出即结束」变成有状态准备流程——**低频→高频立项的产品结构直接参照**。
2. **评分的责任分离与可解释性**（`candidate_score.py`）：LLM 只产结构化事实字段，确定性代码算归一化分 + 九项 `score_breakdown` + 逐项人话理由，配纯函数回归测试。**直接适用于我们简历评分设计**（`knowledge-base/specs/2026-05-20-resume-scoring-design.md`）：可复现、可测试、推荐理由必须引用分项。
3. **面试包 + 简历 lint 禁用写法**：STAR 4-5 行硬约束、「无指标就改写成工程产出表达」的降级话术、三条禁用写法（「熟练掌握」无模块 / 「显著提升」无口径 / 「独立完成」讲不清边界）——可直接做成简历 lint 规则与 AI 改写原则；从简历+JD 生成「面试拷问 Q&A」是零迁移成本的高频延伸。

**别学**：当投递管理参考（无此实体，数据模型得自己设计）；对特定 agent 运行时控件的硬依赖；正则/文件名启发式（我们有真 LLM 管线）。

## 四、汇总：三个行动指向

1. **多模板**（若立项）：留在 LaTeX 层做多套生成风格；学 Resume-Matcher 的数据/样式分离 + 单一 registry + 注册守卫测试；不开 HTML 第二渲染链路。
2. **AI 编辑质量**：把 vibe-resume 的「编辑纪律 + 改后产物断言」和 shushu 的「简历 lint 禁用写法」并入 Coco 的编辑/生成工具链（低成本、随时可做）。
3. **低频→高频立项**：产品结构直接参照 shushu——JD 一等实体、按 JD 组织版本、投递前 readiness 检查表、面试包延伸；评分实现参照其 breakdown + 分项理由模式，与已有简历评分设计衔接。
